# 01. 非機能要件

> **このドキュメントの守備範囲**：性能・セキュリティ・コスト・可用性・拡張性・観測性（要件レベル）・テスト方針。
> **採用ライブラリ・サービス具体名と選定理由**は [05-runtime-stack.md](./05-runtime-stack.md) を参照。
> **観測性の詳細設計**は [04-observability.md](./04-observability.md)、**コンポーネントの責務**は [02-architecture.md](./02-architecture.md) を参照。

---

## セキュリティ（最重要）
- サンドボックスはホストから隔離（→ [04: サンドボックスランナー](./02-architecture.md#サンドボックスランナーgo-ワーカー内で実行)）
- ネットワークアクセス遮断
- ファイルシステム書き込み制限（/tmp のみ、サイズ上限）
- 実行時間制限（例：5秒）、メモリ制限（例：256MB）
- CPU 制限（cgroups）
- 脱獄・リソース枯渇攻撃への対策をドキュメント化
- レート制限：**Sliding Log 方式**を採用
  - 想定規模ではメモリ消費が小さく、正確性とデバッグ容易性を優先
  - 大規模化時は Window Counter 方式への移行を検討
  - 実装ライブラリ・ストレージは [05-runtime-stack: キャッシュ / セッション](./05-runtime-stack.md#キャッシュ--セッション) を参照

## パフォーマンス
- 問題生成：非同期、ユーザーは生成完了を待たない（ジョブキュー）
- 採点レスポンス：通常時 P95 < 3秒
- サンドボックス起動：< 500ms（隔離強化フェーズ以降の目標、→ [04: サンドボックスランナー](./02-architecture.md#サンドボックスランナーgo-ワーカー内で実行)）

## 可用性
- ポートフォリオ用途のため SLA は定めないが、落ちない設計は意識
- ジョブ失敗時のリトライ（最大3回、指数バックオフ）

## コスト
- LLM API 月額上限を設定（例：$30/月）
- キャッシュで同一プロンプトの重複呼び出しを回避
- 低コストモデル → 高品質モデルの段階的利用でコスト最適化（→ [05: コスト最適化](./03-llm-pipeline.md#コスト最適化)）

## 拡張性
- サンドボックスランタイムは差し替え可能な抽象化レイヤを通す（→ [04: サンドボックスランナー](./02-architecture.md#サンドボックスランナーgo-ワーカー内で実行)）
- 対応言語を後から追加できる構造（言語ごとの実行アダプタ）

## 観測性
- 全リクエストにトレース ID
- LLM 呼び出しの入出力・トークン数・レイテンシをログ化
- エラー追跡（→ [06: エラー追跡](./04-observability.md#エラー追跡)）

## テスト
- ユニットテスト：ビジネスロジック
- 統合テスト：サンドボックス実行周り
- E2E：主要ユーザーフロー（問題生成 → 解答 → 採点）
