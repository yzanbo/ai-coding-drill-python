# `03-user-stories.md` 用テンプレート

[03-user-stories.md](./03-user-stories.md) を 0 ベースから作成する時、または**新しいユーザーストーリーを追加する時**（最も頻繁な追記）に使う雛形。

`03-user-stories.md` は **「ペルソナが何をしたくて、なぜしたいのか」** を扱う。マトリクス形式で蓄積していく**成長するドキュメント**。

ペルソナそのものの定義は [`_template-02-personas.md`](./_template-02-personas.md)、機能個別の受け入れ条件は [4-features/_template.md](../4-features/_template.md) の領分。

---

## 使い方

### ケース 1：0 ベースから 03-user-stories.md を作る時

1. 「ファイル全体テンプレ」セクションをコピーして [03-user-stories.md](./03-user-stories.md) に貼り付け
2. ペルソナごとの「ペルソナ別マトリクス」テンプレを必要な数並べる
3. 各マトリクスに「ストーリー単位テンプレ」で行を追加していく

### ケース 2：既存マトリクスにストーリーを追加する時（最頻）

1. 「ストーリー単位テンプレ」セクションをコピー
2. [03-user-stories.md](./03-user-stories.md) の該当ペルソナのテーブルに行を追加
3. `{...}` placeholder を埋める
4. 関連機能（F-XX）へのリンクが正しいことを確認

### ケース 3：新ペルソナの登場でマトリクスごと追加する時

1. まず [02-personas.md](./02-personas.md) にペルソナ定義を追加（[`_template-02-personas.md`](./_template-02-personas.md) 参照）
2. 「ペルソナ別マトリクス」テンプレをコピーして [03-user-stories.md](./03-user-stories.md) に追加
3. ストーリー単位テンプレで行を埋めていく

---

## ストーリー追加のタイミング

- **新機能（[4-features/](../4-features/) に新規 .md 追加）を作る時**：**まずここにストーリーを追記してから** 機能要件詳細を書く（順序を守る）
- 既存機能の利用シーンが拡張された時
- 新ペルソナを追加した時
- 受け入れ条件のレビュー中に「このストーリーが抜けていた」と気付いた時（既存ストーリーと矛盾しないか確認）

## 守備範囲の遵守チェック（書く前に）

- [ ] **観測可能・テスト可能な振る舞い**として書けるか（NG: 「使いやすい」「速い」、OK: 「3 秒以内に結果が表示される」）
- [ ] 動機（So that）が **ペルソナの利用文脈**と整合しているか
- [ ] 該当する機能（F-XX）または横断テーマへのリンクが付いているか
- [ ] 機能個別の**受け入れ条件**を書いていないか（受け入れ条件は [4-features/](../4-features/) 側）

```markdown
> **ペルソナの定義**は [02-personas.md](./02-personas.md)、**機能ごとの受け入れ条件**は [4-features/](../4-features/) 配下の各 .md を参照。

## ストーリー追加時のルール

1. **新しい機能を追加する時**は、まずこのファイルにユーザーストーリーを追記してから [4-features/_template.md](../4-features/_template.md) で機能要件を作成する
2. **新しいペルソナが必要になった時**は [02-personas.md](./02-personas.md) に定義を追加してから本ファイルにストーリーを追加
3. ストーリーは `As a / I want / So that` または **「何をしたいか — なぜ」** の形式で記述
4. **観測可能・テスト可能な振る舞い**として書ける粒度を意識する（具体的な受け入れ条件は features/ で詳細化）
```

## ペルソナ別マトリクステンプレ（H2 セクション 1 つ分）

```markdown
## ペルソナ {識別子 A/B/C/...}：{ペルソナ名}

`As a` {ペルソナ名}
`I want` 〜 `So that` 〜

| {状態 or シナリオ} | ストーリー | 関連機能・横断テーマ |
|---|---|---|
| {ここにストーリー単位テンプレで行を追加していく} | | |

---
```

ペルソナ A（最重要）には「状態」列（ゲスト時 / 認証時 等）が有用。ペルソナ B/C のように 1 つのコンテキストしか持たない場合は「状態」列を省略してもよい：

```markdown
## ペルソナ {識別子}：{ペルソナ名}

`As a` {ペルソナ名}
`I want` 〜 `So that` 〜

| ストーリー | 関連機能・横断テーマ |
|---|---|
| {ここにストーリー行を追加} | |
```

## ストーリー単位テンプレ（マトリクスの 1 行分・最頻使用）

該当ペルソナのテーブルに以下のいずれかの形式で 1 行追加：

### 形式 1（標準・推奨）：「何をしたい — なぜしたい」

```markdown
| {状態 or シナリオ} | {何をしたい} — {なぜしたい・得られる価値} | [F-XX](../4-features/F-XX-...md) |
```

例：
```markdown
| 認証時 | カテゴリ・難易度を選んで新しい問題を生成リクエストしたい — 自分の興味・弱点に応じた練習をしたいから | [F-02](../4-features/F-02-problem-generation.md) |
```

### 形式 2（厳密・複雑なケース用）：As a / I want / So that

複数のペルソナが関わる、または複雑なフローを示す時はブロック形式：

```markdown
> As a {ペルソナ名}
> I want {何をしたい}
> So that {なぜ・得られる価値}
```

これをマトリクスのセル内に書くか、別ブロックとして添える。

## 良いストーリー文の特徴

| ✅ 良い | ❌ 悪い |
|---|---|
| 「ログイン後 2 秒以内にホーム画面が表示される — 待ち時間でモチベーションを失わないため」 | 「速くログインできる — 使いやすいから」 |
| 「過去の解答履歴と弱点カテゴリを確認したい — 進捗と苦手分野を客観的に把握したいから」 | 「履歴を見られる」 |
| 「コードエディタで型診断を受けながら解答を書きたい — 構文ミスをサーバ採点前に潰したいから」 | 「便利なエディタが使える」 |

ポイント：

- **What（何をしたい）と Why（なぜ）のセット**：両方ないと優先度判断ができない
- **動詞は具体的に**：「使う」「見る」より「型診断を受ける」「弱点カテゴリを確認する」
- **時間・回数・数量があれば書く**：「3 秒以内」「7 日間」「5/5 通過」など
- **NG ワード**：「使いやすい」「速い」「便利」「いい感じに」は受け入れ条件に翻訳できない

---

## ストーリーの粒度

ストーリーは **「機能 1 つに対して 1〜3 個」** が目安：

- 1 個：シンプルな機能（ログイン等）
- 2〜3 個：複数の状態・シナリオがある機能（解答送信 → ポーリング → 結果表示）
- 4 個以上：機能を分割した方がよい兆候

具体的な画面遷移・条件分岐は **このファイルではなく [4-features/](../4-features/) に書く**。
