# 03. 非機能要件

## セキュリティ（最重要）
- サンドボックスはホストから隔離（gVisor または Firecracker）
- ネットワークアクセス遮断
- ファイルシステム書き込み制限（/tmp のみ、サイズ上限）
- 実行時間制限（例：5秒）、メモリ制限（例：256MB）
- CPU 制限（cgroups）
- 脱獄・リソース枯渇攻撃への対策をドキュメント化
- レート制限：**Sliding Log 方式**（Redis ZSET）を採用
  - 想定規模ではメモリ消費が小さく、正確性とデバッグ容易性を優先
  - `@nestjs/throttler` + Redis ストレージで実装
  - 大規模化時は Window Counter 方式への移行を検討

## パフォーマンス
- 問題生成：非同期、ユーザーは生成完了を待たない（ジョブキュー）
- 採点レスポンス：通常時 P95 < 3秒
- サンドボックス起動：< 500ms（Firecracker 採用時）

## 可用性
- ポートフォリオ用途のため SLA は定めないが、落ちない設計は意識
- ジョブ失敗時のリトライ（最大3回、指数バックオフ）

## コスト
- LLM API 月額上限を設定（例：$30/月）
- キャッシュで同一プロンプトの重複呼び出しを回避
- Haiku → Opus の段階的利用でコスト最適化

## 拡張性
- サンドボックスランタイムは抽象化（Docker / gVisor / Firecracker を差し替え可能）
- 対応言語を後から追加できる構造（言語ごとの実行アダプタ）

## 観測性
- 全リクエストにトレース ID
- LLM 呼び出しの入出力・トークン数・レイテンシをログ化
- エラー追跡（Sentry）

## テスト
- ユニットテスト：ビジネスロジック
- 統合テスト：サンドボックス実行周り
- E2E：主要ユーザーフロー（問題生成 → 解答 → 採点）
