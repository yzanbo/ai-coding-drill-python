# 08. 開発マイルストーン

ポートフォリオ完成までの段階的な開発計画。各フェーズで「面接で見せられる状態」を意識する。

## Phase 0: 基盤（1週間）
- リポジトリ作成、モノレポ構成決定
- Docker Compose でローカル環境（Postgres, Redis）
- GitHub Actions：lint + test の最小構成
- 簡易 README、アーキテクチャ図の初版

**完了条件**：`docker compose up` で開発環境が立ち上がる

## Phase 1: MVP（2〜3週間）
- **NestJS API 初版**：認証（GitHub OAuth）、問題 CRUD、LLM 呼び出し、ジョブ投入
- 問題生成 API（同期、Haiku のみ）
- **採点ワーカー（Go）初版**：Postgres `jobs` テーブルから `FOR UPDATE SKIP LOCKED` でジョブ取得 → Docker で TS コード実行 → 結果書き戻し
- ジョブテーブル設計とペイロード定義（JSON Schema 管理、TS/Go 両対応の型生成）
- 問題表示・解答・採点の一連フロー
- 最低限のフロントエンド（Next.js + CodeMirror 6）

**完了条件**：ユーザーが問題を生成 → 解答 → Go ワーカーで採点 → 結果表示まで通る

## Phase 2: 品質保証パイプライン（1〜2週間）★ 評価の核
- 非同期ジョブ化（リトライ・DLQ・スタックジョブ回収を実装、→ [04: ジョブキュー](./04_architecture.md#ジョブキューpostgres-select-for-update-skip-locked)）
- **LLM-as-a-Judge 実装**（自前、多軸スコアリング、複数回実行平均）
- **ミューテーションテスト導入**（模範解答を改変しテストの厳密性を検証、→ [07: 品質評価まわりのツール](./07_tech_stack.md#品質評価まわりのツール)）
- モデル段階利用（Haiku → Sonnet）
- プロンプトキャッシュ、Redis キャッシュ
- 構造化出力の厳密化（JSON Schema）

**完了条件**：生成成功率・コストがダッシュボード化されていなくても計測可能

## Phase 3: サンドボックス強化（1週間）★ 評価の核
- gVisor 対応
- ベンチマーク（Docker vs gVisor の起動時間・セキュリティ）
- README にサンドボックス設計の詳細ドキュメント

**完了条件**：ランタイム切り替えが設定で可能、ベンチマーク結果が README に載る

## Phase 4: 観測性（1週間）★ 評価の核
- OpenTelemetry 組み込み
- Grafana ダッシュボード（生成成功率・コスト・レイテンシ）
- Sentry 接続
- 構造化ログ整備

**完了条件**：面接官にダッシュボードを見せられる

## Phase 5: 仕上げ（1週間）
- IaC（Terraform）整備、本番デプロイ
- E2E テスト
- README の完成（設計判断、ベンチマーク、デモ動画）
- ポートフォリオサイトからリンク

**完了条件**：面接官が URL からサービスを触れる

## 任意フェーズ
- **Phase 6**：適応型学習（弱点分析 → 出題）
- **Phase 7**：**Python 評価・分析パイプライン追加**（次バージョンの目玉）
  - 生成済み問題の一括再評価バッチ（Judge プロンプト改善時の回帰テスト）
  - 重複検出（embedding + クラスタリング）、カテゴリ・難易度分布の分析
  - 人間評価 vs LLM Judge の相関分析（Judge の信頼性モニタリング）
  - 学習履歴分析バッチ、Jupyter 可視化レポート
  - RAG による教材準拠問題生成を追加する場合は Ragas 導入検討（LlamaIndex）
- **Phase 8**：採点対象言語の多言語化（Python、Next.js コンポーネント採点など）
- **Phase 9**：Firecracker 対応

## 合計期間目安
MVP〜仕上げまで **約6〜8週間**（専業想定）。兼業なら 2〜3 ヶ月。

## 優先度の考え方
- Phase 2〜4 が**ポートフォリオ評価の核**。ここを厚くする。
- Phase 5 の「触れる状態」と「READMEの質」は必達。
- Phase 6 以降は余力次第。未実装でも README で「次に実装予定」と触れるだけで価値がある。
