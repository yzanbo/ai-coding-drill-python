## 役割別環境構築の構造

Backend（Python / FastAPI）/ Frontend（Next.js / TS）/ Worker（Go）は同じ **環境構築 + 品質ゲート 5 ステップ** パターンで進める：

```
[環境構築（複数ステップに分解）]
  - mise install <ランタイム>      # mise.toml の pin を実体化
  - apps/<name>/ の workspace 初期化 + アプリ雛形 + 静的検査ツールのインストール
  - （Python のみ）DB 基盤 = Docker Compose + ORM + マイグレーション
  - （Go のみ）サンドボックスイメージ Dockerfile スケルトン

[品質ゲート 5 ステップ]
  - mise.toml に <name>:* タスク追記
  - lefthook.yml に <name> 用 pre-commit 追加（静的検査：lint / format / typecheck / knip）
  - lefthook.yml に <name> 用 pre-push 追加（動的検証：テスト / ビルド）
  - GitHub Actions に <name> ジョブ追加（ci-success の needs: に追加）
  - dependabot.yml の対応エコシステムのコメントアウトを解除
```

**hook 役割分担の原則**：

- **pre-commit**：静的検査のみ（速度許容 < 10 秒目安）。ruff / pyright / biome / tsc / knip / gofmt / golangci-lint
- **pre-push**：動的検証（テスト・ビルド、10〜60 秒許容）。pytest / vitest / next build / go test。DB 依存テストは DB 未起動なら **block 方針**（fail させて push をブロック、`docker compose up -d` 後に再 push する運用、graceful skip は採用しない）
- **CI**：上記すべて + 依存衛生（pip-audit / deptry / syncpack / govulncheck）。最終再保険

## 役割別 setup の後段：レイヤ分割フェーズ

各役割の setup が完了したあと、Backend / Frontend / Worker それぞれで**プロジェクト固有のレイヤ分割と import 方向**を確定するフェーズが続く。これは「悩む余地のない基本構造」と異なり、プロジェクトのドメイン特性を反映した設計判断を含むため、役割別環境構築とは別フェーズとして扱う。

具体的には：

- 該当役割の `app/` または `apps/workers/<name>/` 配下にレイヤ別ディレクトリを `__init__.py`（Python の場合）または雛形ファイル（TS / Go 等）付きで配置
- 各レイヤの責務・呼び出しの向き・命名規則を **`.claude/rules/{backend,frontend,worker}.md`** に「実装契約」として固定（Claude が自動 load して読む契約ファイル）
- 各ディレクトリに人間向けの README.md（「とは何か」「機能パターン」「やってはいけないこと」）を配置
- 完了基準：契約ファイルに表 + OK/NG 例があり、各レイヤの README が揃い、既存テストが通る

R1 以降の機能実装はこの契約に従う。詳細手順は同ディレクトリ配下の対応する `<layer>-layers.md` を参照。

## 編集ルール

**フェーズ番号やファイル名が変わっても他ファイルの編集が不要になる書き方をする**。連番・ファイル名を変えただけで複数ファイルの書き換えが発生する運用は追従漏れの温床になるため、プロース中では番号・固有ファイル名を直接書かず、相対的・抽象的な表現を使う。

### r0-setup 内部のファイル名・step 番号

- ❌ 固有ファイル名・step 番号をプロース中に裸書き：「foundation.md で導入する」「backend.md と同じパターン」「step 3 の手順」
- ✅ 相対的・抽象的な表現：「本フェーズで導入する」「r0-setup 配下の全フェーズに適用」「Backend フェーズと同じパターン」「以降の役割別フェーズ」「同ディレクトリ配下の `<phase>.md`」
- Markdown リンク（`[label](./path)`）はファイル名がパス部分に現れるが、リネーム時は grep / IDE で機械的に追従できるので可
- r0-setup 配下のファイルにフェーズ順を示す連番プレフィックス（`01-` / `02-` …）は付けない（順序は [01-roadmap.md](../01-roadmap.md) の R0-X 行で表現する、SSoT 集約）

### R0-X / R1-X 等のロードマップ項目番号への参照

**r0-setup 配下からロードマップ項目番号（R0-X / R1-X / R2-X 等）を裸書きで参照しない**。理由は [01-roadmap.md](../01-roadmap.md) の R0-X / R1-X 番号は開発進行中に並び替え・追加・削除が頻発するため（実際に R0-X も R1-X も再番号化された実績あり）、r0-setup/ から参照すると追従漏れが起こりやすい。01-roadmap.md がロードマップ番号の SSoT で、r0-setup/ は「いつの番号か」より「何のフェーズか」を語ればよい。自フェーズ自身も例外ではない（「R0-4 行の状態列」のような自己参照も避け、「本フェーズに該当する行」と書く）。

- ❌ プロース中の番号裸書き：「R1-3 で実体化」「R1-2 LLM プロバイダ抽象化が本フェーズに依存」「R0-4 行の状態列」「R0-9 では入れない」
- ✅ 抽象的フェーズ表現：「後続の Worker 機能実装」「LLM プロバイダ抽象化フェーズが本フェーズの成果物に依存」「本フェーズに該当する行の状態列」「本フェーズでは入れない」
- ✅ 必要なら [01-roadmap.md](../01-roadmap.md) へのリンクで誘導（リンクは grep で機械追従できるため、項目番号本体を裸書きするより安全）
