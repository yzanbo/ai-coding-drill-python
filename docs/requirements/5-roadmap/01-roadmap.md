# 01. ロードマップとプロダクトバックログ

> **このドキュメントの守備範囲**：プロダクト全体の方向性（ビジョン）、リリース単位のロードマップ、優先度付きプロダクトバックログ、スプリント運用の基準（DoR / DoD）、見積もり方針、リスク。
> **個別機能の受け入れ条件・実装単位は [features/](../4-features/) が SSoT**、**設計判断の履歴は [docs/adr/](../../adr/) が SSoT**。
> アジャイル形式で運用するため、**「Phase X を着手前に固定」ではなく、ビジョンに沿って優先度を継続的に見直す**。

---

## ビジョン（変わらない北極星）

LLM が自動生成したプログラミング問題を、サンドボックスで動作保証してから提供する **TypeScript 練習サービス**を、ポートフォリオとして公開可能な水準で完成させる。

### 成功の定義（リリースレベル）

| 指標 | 目標 |
|---|---|
| 採用担当者がリンク 1 つでサービスを触れる状態 | 必達 |
| README から設計判断（ADR）を辿って論理的に説明できる | 必達 |
| LLM 出力の品質保証パイプラインが多層防御として動作する | 必達 |
| サンドボックスが段階的隔離強化（Docker → gVisor → Firecracker）の道筋を持つ | 達成度を見せる |
| 観測性ダッシュボードで生成成功率・コスト・レイテンシが見える | 必達 |

## ロードマップ（リリース単位）

時系列の大枠。**詳細タスクはバックログで管理**するためここには列挙しない。各リリースの粒度は「面接で見せられる状態」になることを基準にする。

| リリース | 主要アウトカム | 主な対象機能・トピック |
|---|---|---|
| **R0：基盤立ち上げ** | `docker compose up` で開発環境が立ち上がる、CI が動く | モノレポ・DB・Redis・GitHub Actions・補完ツール一式（→ [ADR 0018](../../adr/0018-phase-0-tooling-discipline.md)） |
| **R1：MVP（最小貫通）** | ユーザーが問題を生成 → 解答 → 採点 → 結果表示まで一気通貫で動く | [F-01](../4-features/F-01-github-oauth-auth.md) 〜 [F-05](../4-features/F-05-learning-history.md)（同期/単純構成で OK） |
| **R2：品質保証パイプライン** ★ ポートフォリオ評価の核 | 「LLM 出力を信用しない」設計思想が動作で示せる | LLM-as-a-Judge / ミューテーションテスト / プロンプトキャッシュ / 構造化出力厳密化 / 非同期ジョブ化（リトライ・DLQ・スタックジョブ回収） |
| **R3：サンドボックス強化** ★ ポートフォリオ評価の核 | Docker → gVisor 切替が設定で可能、ベンチマーク結果が README にある | gVisor 対応 / 隔離強化ベンチマーク / セキュリティドキュメント化（→ [ADR 0008](../../adr/0008-disposable-sandbox-container.md)） |
| **R4：観測性** ★ ポートフォリオ評価の核 | 面接官にダッシュボードを見せられる、ログ・トレース・メトリクスが連結 | OpenTelemetry 組み込み / プロセス境界トレース連携（→ [ADR 0017](../../adr/0017-w3c-trace-context-in-job-payload.md)）/ Grafana ダッシュボード / Sentry 接続 / アラート整備 / [F-08](#f-08-管理ダッシュボード) 着手 |
| **R5：仕上げ・公開** | 面接官が URL からサービスを触れる、README に設計判断・ベンチマーク・デモ動画が揃う | IaC（Terraform）/ 本番デプロイ / E2E テスト / README 完成 / ポートフォリオサイトからリンク |

### 任意（バックログ末尾、優先度低）

| リリース | 主要アウトカム |
|---|---|
| R6：適応型学習 | 弱点に基づく問題生成（[F-06](#f-06-適応型出題)）、ヒント機能（[F-07](#f-07-llm-ヒント機能)） |
| R7：Python 分析パイプライン | 一括再評価バッチ・重複検出・Judge 信頼性モニタリング・教材準拠 RAG（次バージョンの目玉） |
| R8：多言語化 | 採点対象言語に Python・React コンポーネント等を追加 |

### 期間目安（Forecast、コミットではない）

| 想定 | R0〜R5 まで |
|---|---|
| 専業 | 約 6〜8 週間 |
| 兼業 | 2〜3 ヶ月 |

→ アジャイルでは「いつまでに何を出すか」は **ベロシティ × 残バックログ** で都度見積もる。期間は固定計画ではなく予測値として扱い、毎スプリント終了時に再評価する。

## スプリント運用

### スプリント長

- **1 週間スプリント**を基本単位とする（兼業の場合は実態に合わせて 2 週間に伸ばしてよい）
- スプリント開始時にバックログから上位を取り、終了時に DoD を確認する

### Definition of Ready（DoR）：バックログ項目がスプリントに入る基準

スプリントに引き入れる前に以下を満たすこと：

- [ ] features/ に該当機能の .md がある（またはこのバックログ項目が独立した技術タスクとして明確）
- [ ] **ユーザーストーリー** または技術タスクの目的が書かれている
- [ ] **受け入れ条件**（テスト可能な粒度）がリストアップされている
- [ ] スコープ外が明記されている
- [ ] 影響範囲（変更ファイル・横断テーマ）が把握されている
- [ ] 必要な ADR があれば起票済み（または起票予定が明示されている）

### Definition of Done（DoD）：スプリント終了時に各項目で満たすこと

- [ ] 受け入れ条件すべてを満たす
- [ ] バックエンド / フロントエンド / ワーカー（必要範囲）の実装が完了している
- [ ] ユニットテスト / 必要に応じて E2E テストが追加されている
- [ ] 型チェック・lint・既存テストが green
- [ ] レビュー済みで PR がマージされている
- [ ] features/ の「ステータス」セクションが更新されている
- [ ] 重要な設計判断があれば ADR が起票・更新されている

### スプリントセレモニー（軽量）

個人開発のため形式的に厳密にやらず、以下を**最低限**実施する：

| セレモニー | 頻度 | 内容 |
|---|---|---|
| プランニング | スプリント開始時 | Now バックログから引き入れる項目を選ぶ、DoR を満たすか確認 |
| デイリー（自己振り返り） | 毎日 | 進捗・ブロッカー・今日の優先順位を 5 分で書き出す（任意） |
| レビュー | スプリント終了時 | 完了項目のデモを README / 動画化、ポートフォリオ素材として残す |
| レトロスペクティブ | スプリント終了時 | 良かったこと・改善点・次スプリントへのアクションを記録（必要なら ADR 化） |

## 見積もり方針

### ポイント制（緩やかに）

ストーリーポイントを採用するが、個人開発ではベロシティの厳密化より **「相対サイズの感覚を養う」** ことが目的：

| ポイント | 目安 |
|---|---|
| 1 | 数時間で終わる小さい変更 |
| 2 | 半日〜1 日 |
| 3 | 1〜2 日（典型的な機能追加） |
| 5 | 数日（横断的影響あり） |
| 8 | 1 週間以上（**スプリント内で 1 つ取れば最大**、要分割検討） |
| 13+ | 大きすぎる、必ず分割する |

### 期間予測

`残ポイント ÷ 平均ベロシティ ≒ 残スプリント数` で都度予測する。**コミットメントとは扱わない**。

## リスクレジスタ

ロードマップを脅かす可能性のある要因と監視・対応方針：

| リスク | 影響 | 監視 / 対応 |
|---|---|---|
| LLM API のコストが想定を超える | R1 以降の生成・採点機能の継続可否に直結 | 日次コストアラート（[04-observability.md](../2-foundation/04-observability.md#アラート)）+ キャッシュ強化 |
| Drizzle v1.0 stable リリース時期が読めない | ORM バージョン選定が不安定化（→ [ADR 0016](../../adr/0016-drizzle-orm-over-prisma.md)） | [drizzle.md](../../../.claude/rules/drizzle.md) のチェックリストで実装着手時に確定 |
| サンドボックス起動時間が想定（200ms）を超過 | スループット・コストへの影響 | R1 終了時にベンチマーク、必要なら R3 を前倒し |
| プロセス境界トレース連携を後回しにすると修正コストが膨大 | R4 で詰む | R1 の最初のジョブ INSERT で一緒に実装する（→ [ADR 0017](../../adr/0017-w3c-trace-context-in-job-payload.md)） |
| 補完ツール（Knip / lefthook 等）を後から入れると整地コストが膨大 | 全期間に影響 | R0 で必ず導入する（→ [ADR 0018](../../adr/0018-phase-0-tooling-discipline.md)） |
| ポートフォリオとしての差別化軸がぼやける | R5 公開後の評価が下がる | R2〜R4 を最優先、R5 で README に設計判断を凝縮 |

## 補足

### 旧「Phase X」表記との対応

旧版（ウォーターフォール的フェーズ計画）からの移行マッピング：

| 旧 Phase | 新リリース | 備考 |
|---|---|---|
| Phase 0 | R0 | 基盤立ち上げ |
| Phase 1 | R1 | MVP |
| Phase 2 | R2 | 品質保証パイプライン |
| Phase 3 | R3 | サンドボックス強化 |
| Phase 4 | R4 | 観測性 |
| Phase 5 | R5 | 仕上げ・公開 |
| Phase 6〜9 | R6〜R9 | 任意の拡張、優先度 Later |

ADR の本文に残る「Phase X」表記は Append-only 原則のため変更しない（リリース指示子として読み替える）。

### このドキュメントの更新タイミング

- **スプリント終了ごと**：完了項目を Now → 完了に移し、Next の見直しを行う
- **リリース完了時**：ロードマップ表の該当行を完了マークに更新
- **新しい設計判断発生時**：関連するバックログ項目に ADR リンクを追加
