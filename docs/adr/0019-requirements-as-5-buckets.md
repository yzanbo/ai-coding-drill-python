# 0019. 要件定義書を「時系列 × 変更頻度」の 5 バケット構造に再編

- **Status**: Accepted
- **Date**: 2026-05-04
- **Decision-makers**: 神保 陽平

## Context（背景・課題）

要件定義書のディレクトリ構造を決める必要がある。当初は以下の二層構造で運用していた：

```
docs/requirements/
├── base/         (10 章の章立て章；機能要件・非機能要件・アーキ・etc を一括)
└── features/     (機能別 .md：F-XX-...md)
```

しかし運用していくうちに以下の問題が顕在化した：

1. **`base/02_functional.md` と `features/F-XX.md` の役割重複**
   - 機能の「俯瞰一覧」と「詳細仕様」が両方に書かれ、SSoT が曖昧
   - 機能追加時に 2 箇所更新する必要がありズレが発生

2. **`base/` 内の章で性質が異なるものが混在**
   - 「変わらないビジョン（01）」と「機能追加で成長する ER 図（09）」が並列
   - git diff のノイズが大きい（変えたくない章と頻繁に触る章が混ざる）

3. **「読む順序」と「書く順序」の混同**
   - 読み手は抽象 → 具体の順で読みたい
   - 書き手は時系列で進めたい（ビジョン → アーキ → 機能 → ロードマップ）
   - 単一の章番号（01〜10）ではこの両者を満たせない

4. **個人開発・アジャイル運用との不整合**
   - 機能追加が頻繁に発生する（features/）
   - スプリントごとに更新されるロードマップが他章と並列だと埋もれる
   - 「触りたくない場所」と「頻繁に触る場所」が物理的に分離されていない

5. **ポートフォリオ評価軸との不整合**
   - 採用担当者は「短時間で全体像を把握 → 興味のある領域に深掘り」したい
   - 単一連番だと**読み始める起点**が不明確

これらは **「ドキュメント構造そのものが設計判断の対象」**であり、ADR として記録する価値がある。

## Decision（決定内容）

要件定義書を **「時系列 × 変更頻度」を同時に表現する 5 バケット構造**に再編。各バケットは以下の役割を持つ：

| バケット | 変更頻度 | 役割 | 主なファイル |
|---|---|---|---|
| **`1-vision/`** | 極小 | プロジェクトビジョン・ペルソナ・ユーザーストーリー | `01-overview.md` / `02-personas.md` / `03-user-stories.md` |
| **`2-foundation/`** | 小 | 変わりにくい全体要件 | 非機能 / アーキ / LLM パイプライン / 観測性 / 実装技術 / 開発フロー |
| **`3-cross-cutting/`** | 中 | 機能追加で成長する横断要件 | ER 図 / API 共通仕様 |
| **`4-features/`** | 大 | 個別機能（F-XX） | F-01〜F-05 |
| **`5-roadmap/`** | 大 | 計画と進捗 | ロードマップ / プロダクトバックログ / DoR / DoD |

ディレクトリ番号（1〜5）は **読む順序（抽象 → 具体）と作成順序（時系列）の両方**を表す。**「書く順序」は `docs/requirements/README.md` に Step 1〜9 として別途明示**し、バケット間を行き来する実態を表現する。

各バケット内で：

- 専用 `_template.md` を配置（`1-vision` のみ実ファイル別に 3 テンプレ）
- `README.md` で役割・更新タイミングを定義
- 章番号は各バケット内で `01` から始まり連番を維持

## Alternatives Considered（検討した代替案）

| 候補 | 概要 | 採用しなかった理由 |
|---|---|---|
| **5 バケット時系列構造** | 時系列 × 変更頻度で物理分離、ディレクトリ番号で順序表現 | （採用） |
| 旧 2 層構造維持（`base/` + `features/`）| シンプル、既存運用そのまま | 機能の俯瞰と詳細の SSoT 曖昧、変更頻度が違う章が混在、ロードマップが埋もれる |
| 1 ファイル巨大要件定義書（`requirements.md` 単一） | 物理シンプル | 部分更新の git diff が読みにくい、目次から目的の節に飛ぶコストが高い、複数人での編集衝突 |
| 機能ベース完全フラット（`features/` のみ、横断は README に記述） | 機能と詳細が完全一致 | 横断テーマ（観測性・LLM パイプライン・アーキ全体）が散逸、SSoT 確立が困難 |
| 4 バケット案（`vision / foundation / features / roadmap`、`cross-cutting` を `foundation` に統合） | バケット数を減らして単純化 | ER 図・API 共通仕様は **「変わりにくい foundation」とは性質が違う**（機能追加で成長する）。混在させると更新頻度が異なるものが同居し git diff が乱雑になる |
| 6 バケット案（4-features を `f-mvp / f-backlog` に分離） | MVP 機能とバックログ機能を物理分離 | バックログは roadmap で管理するべきで、features ディレクトリ自体を分割するのは過剰。F-XX という連番システムも壊れる |
| 章番号を作業順に振り直す（`1-vision / 2-features / 3-foundation / ...`） | 書く順序を物理順序に反映 | 「読む順序」と「書く順序」は本質的に違う。書く順序はバケット間を往復するため番号化に不向き。読み手最適化を優先 |

## Consequences（結果・トレードオフ）

### 得られるもの

- **採用担当者・新規参画者が「上から番号順に読めば理解できる」**：1-vision → 2-foundation → 3-cross-cutting → 4-features → 5-roadmap の自然な順序
- **変更頻度で物理分離**：触りたくない場所（1-vision / 2-foundation）と頻繁に触る場所（4-features / 5-roadmap）が別ディレクトリ → git diff のノイズが減る、PR レビュー時の警戒度が明示できる
- **SSoT が綺麗に確立**：機能個別は 4-features が SSoT、横断方針は 2-foundation / 3-cross-cutting、計画は 5-roadmap、と役割が明確
- **アジャイル運用と整合**：4-features がスプリント実装単位、5-roadmap がバックログ、`/new-requirements` カスタムコマンドが機能別 .md を作る前提と一致
- **テンプレ運用の対称性**：4 バケット（1-vision / 2-foundation / 3-cross-cutting / 4-features）に専用テンプレを配置でき、構造的に SSoT 違反を防げる
- **ポートフォリオ価値**：「ドキュメント設計力」を ADR として体系化している珍しさが採用担当者へのアピールになる

### 失うもの・受容するリスク

- **ディレクトリ階層が 1 段深くなる**：`docs/requirements/2-foundation/05-runtime-stack.md` のようにパスが長い（許容範囲）
- **「読む順序」と「書く順序」の使い分け説明が必要**：[`docs/requirements/README.md`](../requirements/README.md) で両者を別表として明示することで吸収
- **新規バケット追加時に番号が崩れる**：6 番目のバケットが必要になった場合、構造再考が必要。現状の 5 つは予測可能な範囲をカバーしているので当面問題なし
- **クロスリファレンス更新コスト（移行時のみ）**：旧構造（`base/0X_xxx.md`）から新構造への参照を 50+ 箇所更新したが、一度きりの作業
- **書き手はディレクトリ間を行き来する**：書く順序が番号順に進まないことを許容する必要があるが、README の作業順序ガイド（Step 1〜9）で明示

### 将来の見直しトリガー

- バケットが 6 つ以上必要になった場合（番号体系の再考）
- `3-cross-cutting/` の役割が他バケットに吸収できると判明した場合（ER 図を data-only に絞れる等）
- プロジェクト性質が大きく変わった場合（例：チーム開発・大規模化）
- ドキュメント運用の統計的失敗が観測された場合（特定バケットが触られない / 衝突が頻発する等）
