# 0009. 採点コンテナは使い捨て方式（ウォームプール不採用）

- **Status**: Accepted
- **Date**: 2026-04-25
- **Decision-makers**: 神保 陽平

## Context（背景・課題）

採点ワーカーが Docker でサンドボックスコンテナを起動する方式の選択。

- 1 ジョブ = 1 採点
- 採点本体は Vitest 実行で 1〜3 秒程度
- 隔離強度（ユーザーコードを安全に実行）が最優先
- 起動オーバーヘッドが採点全体に占める割合とのバランスを取りたい

## Decision（決定内容）

**ジョブごとに採点コンテナを生成 → 実行 → 破棄する使い捨て方式**を採用する。

## Why（採用理由）

1. **隔離強度を最優先**
   - ユーザーコードを実行する以上、前回実行の影響（メモリリーク・ファイル残存・プロセスゾンビ）が原理的に残らない構造が最も安全
   - ウォームプール方式はリセット不完全時に副作用が残るリスクがあり、隔離強度が一段下がる
2. **設計のシンプルさ**
   - 「毎回まっさらな状態」というモデルが理解しやすく、デバッグ・障害切り分けが容易
   - ウォームプールはプール管理・リセット手順・状態追跡が必要で、規模に対して複雑度が割に合わない
3. **採点本体の所要時間に対するオーバーヘッドが許容範囲**
   - 起動 ~200ms（段階 1 / Docker での想定実測）に対し採点本体は 1〜3 秒、オーバーヘッドは全体の 5〜15% で UX 上問題にならない
   - 隔離強化段階 2 以降でも上限 500ms を維持する目標 → SSoT は [01-non-functional.md: パフォーマンス](../requirements/2-foundation/01-non-functional.md#パフォーマンス)
   - MVP 規模（数百ジョブ/日）では起動コストの最適化より隔離強度を優先する判断が合理的
4. **デプロイ・再起動の安全性**
   - in-flight ジョブが各コンテナに閉じ込められているため、ワーカー再起動・ローリングデプロイで状態が漏れない
5. **R3 / R9 の高速化ロードマップを残せる**
   - スループットが問題化した場合は gVisor（R3）→ Firecracker（R9）の段階的高速化で対応可能
   - 現時点で先取りせず、必要になった段階で適切な方式を選べる構造を維持

## Alternatives Considered（検討した代替案）

| 候補 | 概要 | 採用しなかった理由 |
|---|---|---|
| 使い捨てコンテナ | ジョブ毎に作成・破棄 | （採用） |
| ウォームプール | 一定数のコンテナを起動済みにし、中身だけリセット | リセット不完全時の影響残存リスク、設計が複雑化、隔離強度が一段下がる、現規模ではメリットが小さい |
| Firecracker microVM | 軽量 VM、~100ms 起動 | R9 のロードマップ、MVP では不要 |
| gVisor（runsc） | システムコール隔離強化 | R3 のロードマップ、MVP では Docker 標準で十分 |

## Consequences（結果・トレードオフ）

### 得られるもの
- 前回実行の影響が原理的に残らない強い隔離保証（メモリリーク・ファイル残存・プロセスゾンビ皆無）
- 設計がシンプル、デバッグが楽（毎回まっさらな状態）
- ワーカー再起動・デプロイが安全（in-flight ジョブが少ない）

### 失うもの・受容するリスク
- 起動オーバーヘッド ~200ms（採点本体の 5〜15%）が常に発生
- 高負荷時のスループット上限がコンテナ起動速度に依存

### 将来の見直しトリガー
- スループットが問題化した場合は以下を順に検討：
  1. ウォームプール方式（中身だけリセット）
  2. gVisor で起動高速化 + 隔離強化（R3）
  3. Firecracker microVM（R9）
