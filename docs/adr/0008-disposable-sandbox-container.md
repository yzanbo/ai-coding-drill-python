# 0008. 採点コンテナは使い捨て方式（ウォームプール不採用）

- **Status**: Accepted
- **Date**: 2026-04-25
- **Decision-makers**: 神保 陽平

## Context（背景・課題）

採点ワーカーが Docker でサンドボックスコンテナを起動する方式の選択。

- 1 ジョブ = 1 採点
- 採点本体は Vitest 実行で 1〜3 秒程度
- 隔離強度（ユーザーコードを安全に実行）が最優先
- 起動オーバーヘッドが採点全体に占める割合とのバランスを取りたい

## Decision（決定内容）

**ジョブごとに採点コンテナを生成 → 実行 → 破棄する使い捨て方式**を採用する。

## Alternatives Considered（検討した代替案）

| 候補 | 概要 | 採用しなかった理由 |
|---|---|---|
| 使い捨てコンテナ | ジョブ毎に作成・破棄 | （採用） |
| ウォームプール | 一定数のコンテナを起動済みにし、中身だけリセット | リセット不完全時の影響残存リスク、設計が複雑化、隔離強度が一段下がる、現規模ではメリットが小さい |
| Firecracker microVM | 軽量 VM、~100ms 起動 | Phase 3 のロードマップ、MVP では不要 |
| gVisor（runsc） | システムコール隔離強化 | Phase 2 のロードマップ、MVP では Docker 標準で十分 |

## Consequences（結果・トレードオフ）

### 得られるもの
- 前回実行の影響が原理的に残らない強い隔離保証（メモリリーク・ファイル残存・プロセスゾンビ皆無）
- 設計がシンプル、デバッグが楽（毎回まっさらな状態）
- ワーカー再起動・デプロイが安全（in-flight ジョブが少ない）

### 失うもの・受容するリスク
- 起動オーバーヘッド ~200ms（採点本体の 5〜15%）が常に発生
- 高負荷時のスループット上限がコンテナ起動速度に依存

### 将来の見直しトリガー
- スループットが問題化した場合は以下を順に検討：
  1. ウォームプール方式（中身だけリセット）
  2. gVisor で起動高速化 + 隔離強化（Phase 2）
  3. Firecracker microVM（Phase 3）
