# レビュー指摘への対応（純メリット仕分け + 実装 + 要件整合）

直前の `/pr-review-by-commit` 出力、または会話文脈にある指摘リストを入力として、**実装で潰すもの / 潰さないもの** を純メリット判断で仕分けながら commit を分けて修正する。最後に要件 ↔ 実装の整合性をチェックし、必要なら要件側の更新も提案する。

## このスキルの方針

- **レビュー指摘を無条件には対応しない**：1 件ずつ「純粋なメリット / デメリット」で対応可否を判断する
- **工数を判断軸に入れない**：工数の記載は可。ただし「半日かかるから対応不要」のような仕分けは禁止
- **スコープ判定を別軸として持つ**：「スコープ外でも対応すべき」「スコープ内でも対応不要」が両方あり得る
- **コミットは論理単位で分ける**：1 修正 1 commit に固執しすぎず、関連の強いものはまとめる。逆に無関係な修正を 1 commit に詰め込まない
- **実装中に気付いた追加指摘も同じ基準で取り込む**：対応する場合は同じ仕分けプロセスを通す
- **最終ステップで要件 ↔ 実装の整合性をチェック**：実装側を変えたことで生じた drift を発見し、「要件を直すべき」or「実装を直すべき」を純メリットで提案する
- **最後にレポート**：対応したもの / 対応しなかったもの（理由付き） / 追加で見つけたもの / 要件側の追加更新提案、を整理して出す

## 手順

### 0. 入力の確定

- 引数があればそれを指摘リストとして解釈
- 引数がなければ会話文脈の直前 `/pr-review-by-commit` 出力 or 指摘リストを拾う
- 指摘が見つからない場合はユーザーに「どの指摘リストか」を質問

### 0.5. コード再分析（事前仕分けの前に必ず実施）

仕分けに入る前に、**指摘リストを鵜呑みにせず実コードを読み直す**。レビューはレビュアーの誤読 / 過去版に対する指摘 / 既に別 commit で解消済みの可能性があるため、事前検証なしに仕分けると「存在しない問題に対応する」「既に解決済みのものを再修正する」事故が起きる。

**コード読み込みは ultrathink モードで徹底的に行う**：表面的な行確認で済ませず、call graph・データフロー・暗黙の前提・周辺ファイルとの関係まで踏み込んで読む。仕分けの精度はここで読み込んだ深さに比例するため、時間を惜しまない。

実施内容：

- 指摘が指す **ファイル / 行を実際に開いて読む**（リンクだけ信じない）
- **指摘の前提が現在のコードで成立しているか**を確認（リネーム / 削除 / 後続 commit での修正に注意）
- **周辺コードも合わせて読む**：指摘単独では見えない依存・暗黙の制約・既存 workaround を把握する
- レビュアーの誤読が疑われる時は「指摘の前提が現状コードと一致しない」と仕分け表に明記する
- **再分析中に新たな課題を発見したら、必ず事前仕分けに追加する**（レビューに無い指摘でも、コードを読んで気付いたものは仕分け表の対象。「レビューにないから対応しない」と判断を後送りしない）

再分析の結果を踏まえて初めて §1 の仕分けに進む。

### 1. 事前仕分け（実装着手前）

実装前に **全指摘を一度通して** 仕分け表を作る。**「対応する」「対応しない」のどちらも含めて全件**を再度俯瞰し直すこと（レビュー時点の判断をそのまま流用せず、§0.5 のコード再分析を踏まえて再評価する）。実装中に判断が変わったら都度更新する。

#### 仕分けの観点

各指摘に対して以下を**書き出す**：

| 観点 | 内容 |
|---|---|
| メリット | 機能 / 安全性 / 保守性 / 整合性がどう上がるか |
| デメリット | 複雑度増加 / コード量増加 / 学習コスト / トレードオフ |
| スコープ | 本 PR 内 / 別 PR / 別 Issue のどれが妥当か |
| 判定 | 対応する / 対応しない |
| 判定理由 | 純メリット視点の 1 行根拠 |

**判断基準**：工数無視。純粋なメリット / デメリットだけで決め、コードをしっかり読み込んで判断する（リンク / 記憶だけで仕分けない）。

判定パターンの典型：

- **対応する**：メリット明確 + デメリット軽微（例：セキュリティ重大 / 1 行修正）
- **対応する（但しスコープ外）**：メリット明確だが本 PR で対応するとスコープ拡大しすぎる。**フォロー PR / Issue で対応** と明示
- **対応しない（YAGNI）**：将来必要になるかもしれないが現状で根拠が薄い
- **対応しない（受容済みトレードオフ）**：ADR / 規約で既に judgment 済み
- **対応しない（過去 commit / retroactive 不可）**：履歴系の指摘
- **対応しない（プロジェクト方針整合）**：CLAUDE.md / MEMORY.md / `.claude/rules/` の方針通り
- **対応しない（別 PR で対応済み / 確定）**：ユーザー指示で別 PR スコープ確定済み

#### 出力例

```markdown
## 対応方針（先に判定）

**対応する**
- (40) sid 平文ログ → 秘密情報漏洩防止、デメリット皆無
- (4)(5) 本番 default 値の安全装置 → 事故防止、validator 追加コスト極小
- ...

**対応しない（理由付き）**
- (76) テストゼロ → ユーザー指示で別 PR にスコープ分離
- (10) Redis 接続パラメータ明示 → MVP 規模では premature optimization
- (75) CSRF exempt 属性ベース化 → 現状 2 パスのみ、YAGNI
- ...
```

### 2. コミット分割計画

仕分け後の「対応する」群を **論理単位** でまとめて commit 計画を立てる：

| 観点 | 例 |
|---|---|
| 同じテーマでまとめる | セキュリティ修正 3 件 → 1 commit |
| ファイル別で分けすぎない | 関連の強い 2 ファイル修正は 1 commit |
| 大きな refactor は単独 | 共有 client lifespan 化、state レコード再設計 等 |
| migration は実装と同 commit | model 変更 + migration 生成を 1 commit |
| docs drift は単独 | 規約ファイル更新だけ別 commit |

順序の原則：
- **依存層 → 利用層**：基盤層（core/）の変更を先、上位（services/, routers/）を後
- **互換性のある修正を先**：型変更等を伴う修正は順序を依存に従って並べる
- **生成 artifact を最後**：openapi.json / 生成型は他の変更が落ち着いた後に再生成

### 3. 実装ループ（1 commit ずつ）

各 commit について以下を回す：

1. **対象の指摘を確認**：仕分け表で「対応する」と判断したもの
2. **Edit でコード修正**：可能なら `Edit` で最小差分、新規ファイルは `Write`
3. **lint / typecheck / test を回す**：
   - Backend: `mise run api:lint` / `mise run api:typecheck` / `mise run api:test`
   - Frontend: `mise run web:lint` / `mise run web:typecheck` / `mise run web:test`
   - Worker: `mise run worker:<name>:lint` / `mise run worker:<name>:test`
4. **drift 発生時は artifact 再生成**：OpenAPI / job-schemas / Hey API
5. **git add → git commit**：CLAUDE.md コミットメッセージ規約に従う（WHAT / WHY / HOW / 影響範囲、日本語、AI 生成文言禁止）
6. **lefthook の pre-commit / commit-msg をパス**：失敗したら fix し直して **新規 commit**（amend は CLAUDE.md で禁止）

### 4. 実装中の追加指摘 / 仕分け再判断

実装ループは **新規発見 + 既存判断の見直し** の両方を扱う。事前仕分けは凍結された決定ではなく、実コードを触ることで前提が変わったら都度更新する。

#### 4-A. 新しい問題に気付いた場合

実装を進める中で気付いた問題は、その都度同じ仕分けプロセスを通す：

- **メリット / デメリット / スコープ / 判定 / 理由** を 1 件分書く
- 対応する場合は現在の commit に乗せるか、別 commit を増やすかを判断
  - 同じテーマなら現在の commit に乗せる
  - 無関係なテーマなら commit を分ける
- 対応しない場合は最終レポートの「対応しなかったもの」リストに加える

例：
> 実装中に「`Settings._check_production_safety` を入れたら APP_ENV フィールドも追加すべき」と気付いた → 同じ commit に乗せる（同テーマ）

#### 4-B. 既存判断の再判断（適宜実施）

実装中に **「対応する」と決めた指摘が実は不要だった / 「対応しない」と決めた指摘が実は対応すべきだった** と気付くことがある。事前仕分けの結論を固定せず、以下のタイミングで再判断する：

- **隣接コードを編集した瞬間**：周辺の構造が見えて「ついでに直す方が安い」「分離した方が clean」が判明する
- **lint / typecheck / test が想定外に通る or 落ちる**：前提が崩れた可能性。仕分けの根拠を見直す
- **commit 計画通りに分割できない**：論理単位がズレている。仕分け or 順序を組み直す
- **同じテーマの修正をしている時**：「対応しない」と決めた隣接指摘が、ついでなら純メリットで対応すべきに反転することがある

再判断した場合は、最終レポートの「対応した指摘」「対応しなかった指摘」のどちらかに **再判断の根拠と一緒に記載**する（仕分け表を黙って書き換えない、判断ログを残す）。

例：
> 事前仕分けで「対応しない」とした (27) Playwright cache を実装中の CI ファイル編集中に再判断 → 同じ ci.yml に 1 行追加で済むと判明 → 「対応する」に反転して同 commit に追加

### 5. 要件 ↔ 実装の整合性チェック

**実装を一通り終えた後、最終ステップとして** 必ず実施する。実装側を変えたことで要件側に drift が生まれていないかを洗う。

#### チェック対象

1. **受け入れ条件**（`docs/requirements/4-features/<name>.md` §受け入れ条件）
2. **ビジネスルール**（§ビジネスルール / §N.1）
3. **API / 画面 / フロー仕様**（§N.4 / OpenAPI export）
4. **バリデーション節**（§N.5）
5. **ステータス節**
6. **ADR**
7. **`.claude/rules/`**
8. **CLAUDE.md / MEMORY.md**

#### drift 発見時の判断

各 drift について **「実装を直す」vs「要件を直す」** を純メリットで提案する：

| パターン | 判断 |
|---|---|
| 実装側を要件に揃えるとセキュリティ / 機能 / 保守性が後退 | **要件側を更新** |
| 要件側を実装に揃えると仕様が緩む / 機能要件が消える | **実装を直す** |
| どちらも同等の純メリット | **要件側を更新**（実装が SSoT 原則） |
| 機械的検証ルール（長さ / 型 / 正規表現） | **要件側に書かない**（Pydantic / Zod / DB が SSoT、要件は業務ルールのみ） |
| 運用 / インフラ要件で要件側に未記載 | `2-foundation/01-non-functional.md` 等に追記 |
| 機能要件で要件側に未記載 | `4-features/<name>.md` に追記 |

#### 出力例

```markdown
### drift 4: APP_ENV による本番安全装置の存在が要件に出てこない

- **観察**：本 PR で `APP_ENV=production` 時の起動拒否 validator を追加したが、要件 .md には記載なし
- **判断**：これは **運用 / インフラ要件** で、`01-non-functional.md` のセキュリティ節に書く性質
- **修正提案**：`01-non-functional.md §セキュリティ` に 1 行追記が純メリット。要件側を更新すべき。実装は維持
```

#### 要件側の追加更新を本 PR でやるか別 PR でやるか

ユーザーに確認するか、PR スコープに合うかで判断：

- **本 PR でやる**：認証 PR で `01-non-functional.md` の認証関連 1 行追記など、強く関連する場合
- **別 PR でやる**：完全に別ドメインの要件変更、PR が肥大化する場合

判断に迷ったらユーザーに「このまま PR 内で要件を直すか / 別 PR にするか」を聞く。

### 6. 最終レポート

実装完了後、以下の構造でレポートを出す：

```markdown
## 最終サマリ

### 対応した指摘（X 件、Y commits）

| # | 指摘 | commit | 純メリット判断根拠 |
|---|---|---|---|
| (40) | sid 平文ログ | `d97dbd4` | ログ閲覧者によるセッション乗っ取り防止。デメリット皆無 |
| ... |

### 対応しなかった指摘（理由付き）

| # | 指摘 | 理由 |
|---|---|---|
| (76) | テストゼロ | ユーザー指示で別 PR 対応とスコープ分離 |
| (10) | Redis 接続パラメータ | MVP 規模では premature optimization |
| ... |

### 実装中に追加で対応した指摘

- 仕分けプロセスを通した結果、追加で対応したもの（理由含む）

## 要件 ↔ 実装 整合チェック

### 受け入れ条件 ↔ 実装
（表形式で各受け入れ条件と実装位置を突き合わせ）

### ビジネスルール ↔ 実装
（同上）

### ADR ↔ 実装
（同上）

### 検出した drift と「実装 / 要件どちらを直すべきか」

drift X: <内容>
- 観察：...
- 判断：...
- 修正提案：実装を直す / 要件を直す / 修正不要、の判断と根拠

### 結論
- 本 PR で追加対応する drift：N 件
- 別 PR / Issue でフォローする drift：M 件
- 修正不要（設計判断 OK）：K 件
```

## やらないこと

- **無条件に全指摘を潰さない**：仕分けプロセスを必ず通す
- **「とりあえず全部対応」「とりあえず全部 skip」しない**：1 件ずつ理由を持つ
- **工数を仕分け軸に使わない**：「時間がない」で skip するのは禁止（純メリットのみ）
- **要件整合チェックをスキップしない**：実装を変えた以上、必ず drift を洗う
- **要件側を勝手に全部書き換えない**：要件側の修正は提案ベース、実施可否はユーザー確認またはスコープ明示
- **AI 生成文言を入れない**（CLAUDE.md「絶対ルール」）
- **commit を amend / 強制 push しない**（CLAUDE.md Git 操作禁止事項）
- **`git add -A` / `git add .` を使わない**：明示パスで add する

## 関連

- 対のスキル：`/pr-review-by-commit` でレビューを生成 → `/pr-review-respond` で対応
- 自分のブランチで PR を作る前にレビュー → 対応をループする使い方も可能（セルフレビュー）
- 要件側の更新を別 PR でやる場合は `/update-requirements` スキルへ繋ぐ
