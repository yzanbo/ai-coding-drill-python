# ユーザーマニュアルの整合性検証

`/update-documents` で生成したマニュアルの記載内容と、実際のアプリケーションの動作・画面を比較し、差異があれば修正の是非をユーザーに確認する。

引数 `$ARGUMENTS` を対象ドキュメントの種別として解釈する。

## 引数の解釈

| 引数 | 対象 |
|---|---|
| （なし / 空） | ユーザー向けマニュアル + 管理者向けマニュアル（両方） |
| `user` | ユーザー向けマニュアルのみ |
| `admin` | 管理者向けマニュアルのみ |

## 検証対象のマニュアル

- ユーザー向け：`docs/manuals/user/html/index.html`
- 管理者向け：`docs/manuals/admin/html/index.html`

マニュアルが存在しない場合は、先に `/update-documents` で作成することを提案する。

## 手順

### 1. マニュアルの読み込みと情報抽出

対象マニュアルの HTML を読み込み、以下の情報を抽出する：

- 記載されている画面の一覧と URL
- 各画面の操作手順・説明文
- 入力項目の説明（項目名、必須/任意、バリデーションルール等）
- スクリーンショットのファイル一覧
- 仕様の数値（タイムアウト 5 秒、メモリ 256MB 等）

### 2. アプリケーションとの比較

以下の観点で実際のアプリケーションと突合する：

#### 2-1. 画面構成の差異

- フロントエンドのルーティング（`apps/web/src/app/(routing)/` 配下）を確認する
- マニュアルに記載があるがアプリに存在しない画面がないか
- アプリに存在するがマニュアルに記載がない画面がないか

#### 2-2. 画面の表示内容の差異

- Playwright MCP を使ってローカル環境（`http://localhost:3000`）の各画面にアクセスする
- 認証が必要な画面はテストアカウントでログインしてから確認する
- 画面上のラベル、ボタン名、メニュー項目がマニュアルの記載と一致するか確認する

#### 2-3. 入力項目・バリデーションの差異

- 機能要件（`docs/requirements/4-features/`）およびバックエンドのバリデーションスキーマを確認する
- マニュアルに記載された入力項目（項目名、必須/任意、入力制約）が実装と一致するか確認する
- フロントの Zod スキーマ（`apps/web/src/lib/validation/`）と API の DTO（`apps/api/src/<feature>/dto/`）の両方をチェックする

#### 2-4. API・機能の差異

- バックエンドのコントローラ・サービスを確認し、マニュアルに記載された機能が実装と一致するか確認する
- 新規追加・削除された機能がマニュアルに反映されているか確認する
- Swagger UI（`http://localhost:3001/api/docs`）で API スキーマと突合する

#### 2-5. 採点まわりの仕様の差異

- マニュアルに記載されているサンドボックスの制約値（タイムアウト・メモリ・CPU 上限）が実装と一致するか確認する
- 採点ワーカー（`apps/grading-worker/internal/sandbox/`）の Docker クライアント設定と照合する
- 対応言語・カテゴリの一覧が実装と一致するか確認する

#### 2-6. スクリーンショットの鮮度

- 既存のスクリーンショットと現在の画面を目視比較する
- レイアウトや UI コンポーネントに明らかな変更がある場合は差異として報告する

### 3. 差異レポートの作成

検出した差異を以下の形式でユーザーに報告する：

```
## 検証結果

### 差異あり（要確認）

1. **[画面名/セクション名]** — [差異の概要]
   - マニュアル：[マニュアルの記載内容]
   - 実際：[アプリケーションの実際の状態]
   - 推奨対応：[マニュアル修正 / アプリ修正 / 確認必要]

### 差異なし
- [確認済み画面/セクションの一覧]
```

### 4. 修正の実行

差異ごとにユーザーに修正の是非を確認する：

- **マニュアルを修正する場合**：対象の HTML ファイルを直接編集し、PDF も再生成する
- **アプリケーションを修正する場合**：該当するフロントエンド/バックエンド/ワーカーのコードを修正する
- **要件のみ修正する場合**：機能要件 .md とベース要件を更新する（実装は別途）
- **対応しない場合**：スキップして次の差異に進む

修正する場合は 1 件ずつユーザーの承認を得てから実行する。まとめて修正しない。

### 5. 完了報告

- 検出した差異の総数
- 修正した件数（マニュアル修正 / アプリ修正 / 要件修正の内訳）
- スキップした件数
- 再生成した PDF のファイルパス（マニュアル修正があった場合）
