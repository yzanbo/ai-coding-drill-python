# ADR / 要件定義書 / 補助ドキュメントの SSoT ポリシー

ADR・要件定義書・`.claude/CLAUDE.md`・README・CONTRIBUTING.md・このディレクトリ自身（`.claude/rules/`）を編集する時に従う。

---

## 1. SSoT を分裂させない

同じ規約を 2 箇所以上にフル転記しない。1 箇所を「正」とし、他はリンクで誘導する。例外は §3（`.claude/` 内）のみ。

---

## 2. ADR の Decision は 2 通り。型を見極めて書き分ける

### 戦略判断型：Decision は ADR 本文に書く

一度決めたら基本動かない判断。動かす時は別 ADR で `Superseded`。例：AWS 単独 / Drizzle 採用 / Postgres ジョブキュー / LLM プロバイダ抽象化 / 使い捨てサンドボックス。

→ Decision セクションに判断本体・詳細表・運用ルールを完結させる。

### 運用ルール型：Decision は要件定義書または機械強制設定に SSoT を置く

時間で進化する規約。表が増える / フックが増える / 設定値が育つ。例：補完ツール構成 / 設定ファイル形式優先順位 / syncpack ルールセット / commit scope 一覧 / ci-success umbrella。

→ Decision セクションは以下のテンプレートに従う：

```markdown
## Decision（決定内容）

<採用したものを 1〜2 文で>。

**運用詳細の SSoT は [<要件定義書ファイル>: <セクション名>](<相対パス>) を参照**（運用ルール型 ADR、→ `.claude/rules/adr-and-docs-ssot.md` §2）。本 ADR は採用根拠（§Why）と代替案（§Alternatives Considered）を扱う。
```

ADR 本文には「なぜ・代替案・見直しトリガー」だけを残し、表・運用ルールは要件定義書 / 機械強制設定（`commitlint.config.ts` / `.syncpackrc.ts` 等）にだけ書く。

### 判定基準

「**この決定は時間とともに進化していくか / 凍結するか**」で判定する。進化するなら運用ルール型、凍結するなら戦略判断型。

### 要件定義書側（運用ルール型の受け先）の責務

1. 対応セクションを設ける（例：「フック × チェック × CI 対応表」「コミットメッセージ規約」）
2. セクション末に「採用根拠は [ADR 00XX] を参照」を必ず置く（双方向リンク）
3. 機械強制設定ファイルが真の SSoT になる場合は、要件定義書セクションは概観のみとし、詳細は設定ファイルを参照する

---

## 3. `.claude/` 内の特例（直接転記を許容）

`.claude/` 配下のファイル（`CLAUDE.md` + `rules/*.md`）は Claude Code が**自動でコンテキストに読み込む**：

- `CLAUDE.md`：毎セッション読み込まれる
- `rules/*.md`：frontmatter `paths:` にマッチするファイルを編集する時に読み込まれる

Claude はリンクを必ず辿るとは限らないため、**Claude が日常的に従うべき規約は `.claude/` 内のファイルに直接転記してよい**（§1 の例外）。

### 直接転記する基準

- 適用する：短い規約・表・列挙（コミット scope 表、ブランチ命名、設定ファイル形式の優先順位、レイヤ別実装ルール 等）
- 適用しない：長文の前提原則、採用根拠、代替案、フローチャート、トレードオフ議論 — これらは要件定義書 / ADR にだけ書く

### 転記時の必須事項

- 該当セクション冒頭に「**SSoT は <要件定義書 X.md / 設定ファイル>、本セクションは Claude が直接読む用の縮約版（SSoT 更新時はここも合わせて更新する）**」と明記
- SSoT を更新したら `.claude/` 内のミラーも同じ PR で更新する（差分が一致することを PR レビューで人間が確認）

### `.claude/` 内ファイル間の重複は避ける

`.claude/CLAUDE.md` と `.claude/rules/*.md` の間でも同じ規約をフル転記しない。`paths:` スコープが違うため Claude が両方を読むタイミングは異なるが、人間メンテナンスのコストは増えるため：

- **CLAUDE.md**：プロジェクト全体に効かせたい横断規約（コミット形式、ブランチ命名、Git 操作禁止事項 等）
- **rules/*.md**：特定パスでだけ load されればよいレイヤ別ルール（`backend.md` / `frontend.md` / `worker.md` / 本ファイル等）

役割が重なる場合は片方を「正」とし、他方は短いポインタにする（例：本ファイルが ADR / docs SSoT ポリシーの「正」、`requirements-docs.md` §2A はポインタ）。

---

## 4. README / CONTRIBUTING.md / その他の人間向けドキュメント

人間はリンクを辿れる前提。**同じ規約をフル転記しない**。要約を載せる場合も「詳細は <要件定義書ファイル> を参照」のリンクを必ず添える。CLAUDE.md 特例（§3）は適用しない。

---

## 5. `.claude/` 内ファイル（CLAUDE.md / rules/*.md）の書き方（メタルール）

`.claude/CLAUDE.md` と `.claude/rules/*.md`（このファイル含む）を書く・更新する時の基準：

### ルール本体は直接記載する（リンクで逃さない）

- ルールを「他ファイルに書いてある」とリンクで誘導しない。**Claude はリンクを必ず辿るとは限らない**ため、ルール本体は load されるそのファイル内に直接書く
- リンクを使ってよいのは以下に限定：
  - **採用根拠 / 設計判断**：「採用根拠は [ADR 00XX] を参照」（ADR は読まなくても規約には従える）
  - **実装 SSoT のコードファイル**：「機械強制 SSoT は [`commitlint.config.ts`](...)」
  - **より長い解説 / 具体例 / フローチャート**：「詳細フローチャートは [<要件定義書>] を参照」（基本ルールはこのファイルで読み切れる前提）

### Claude が読み取りやすい構造にする

- 見出しで区切り、トピックを明示する
- 規約は**表 / 箇条書き**で列挙する。長文の散文は避ける
- 適用テンプレートはそのまま貼れる**コードブロック**で示す
- 1 ルールにつき動詞（「〜する」「〜しない」）が含まれる 1 文を基準とし、根拠は短く後置する

### `.claude/` 内ファイル間の役割分担を明確にする

複数の rules が同じ paths にマッチして両方 load される場合は、役割分担で重複を避ける：

- **`.claude/CLAUDE.md`**：プロジェクト全体に効かせたい横断規約（コミット形式 / ブランチ命名 / Git 操作禁止 / 設定ファイル形式の優先順位 / コード品質ツール一覧 等）
- **`.claude/rules/adr-and-docs-ssot.md`**（本ファイル）：ADR / 要件定義書 / `.claude/` 内ファイルの SSoT ポリシー
- **`.claude/rules/requirements-docs.md`**：要件定義書ディレクトリ（`docs/requirements/`）内部の記述ルール（5 バケット構造 / Mermaid 規約 / リンクの貼り方 等）
- **`.claude/rules/{backend,frontend,worker,drizzle,prompts}.md`**：レイヤ別実装ルール

役割が重なるトピックが出てきたら、片方を「正」とし、他方は **§3 の例外条件を満たす場合のみ** 直接転記する。基本は 1 箇所だけに書く。
