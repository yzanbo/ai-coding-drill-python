# 要件定義書（`docs/requirements/base/`）の記述ルール

## 1. 重複記述の禁止

- 同じ情報を複数のファイルに重複して書かない。
- 同じ事柄を 2 箇所で説明する必要が生じた場合は、**どちらか 1 箇所を「正」とし、他方からはその箇所へリンクする**。
- 修正時に「片方だけ更新して片方が古いまま」になる事故を防ぐため、内容の単一情報源（Single Source of Truth）を徹底する。

## 2. ファイルごとの守備範囲

各ファイルの責務を分け、重複を構造的に防ぐ。

| ファイル | 担当範囲 |
|---|---|
| `01_overview.md` | プロジェクト概要、ゴール、ターゲット、ロードマップの要約 |
| `02_functional.md` | 機能要件（What を作るか） |
| `03_non_functional.md` | 非機能要件（性能・セキュリティ・コスト・可用性等） |
| `04_architecture.md` | システム全体の論理構造、コンポーネントの責務、データ・ジョブの流れ、運用作法 |
| `05_llm_pipeline.md` | LLM 生成・評価パイプラインの設計、品質評価の多層防御 |
| `06_observability.md` | 観測性（ログ・トレース・メトリクス・アラート） |
| `07_tech_stack.md` | 採用技術（What）+ 選定理由（Why）+ ライブラリ・サービスの具体名 + コスト試算 |
| `08_milestones.md` | フェーズごとの開発計画と完了条件 |
| `09_data_model.md` | ER 図、テーブル定義、ジョブペイロード JSON Schema、インデックス設計、命名規則 |
| `10_api_spec.md` | 主要エンドポイント、認証、エラーフォーマット、OpenAPI 方針 |

### 関連ディレクトリ
- `docs/adr/` — 重要な技術・設計決定の履歴（ADR）。1 決定 1 ファイル、Append-only。新規決定が発生したら `docs/adr/template.md` を元に追加する。

### 04 と 07 の使い分け（最も重複しやすい）

- **04**：「何のコンポーネントが、どんな責務を持ち、どう連携するか」を書く。**ライブラリ名・サービス名・選定理由は書かない**。
- **07**：「各レイヤで具体的にどの技術を採用し、なぜそれを選んだか」を書く。**コンポーネントの責務やデータフローは書かない**。

## 3. リンクの貼り方

- 関連情報が他ファイルにある場合、該当セクションへ**アンカーリンク**で飛ばす。
- リンク先は GitHub Flavored Markdown のスラグルール（小文字化、空白はハイフン、記号は除去）に従う。
- 形式：`→ <短い導入> は [<ファイル名>: <セクション名>](./<ファイル名>#<アンカー>)`
- 例：
  - `→ 採用フレームワーク・ライブラリの詳細は [07_tech_stack.md: バックエンド API](./07_tech_stack.md#バックエンド-api-nestjs--typescript)`
  - `→ コンポーネントの責務は [04_architecture.md: Backend API](./04_architecture.md#backend-apinestjs)`

## 4. 各ファイル冒頭に守備範囲を明記

- 各 `.md` ファイルの先頭に、**自ファイルの守備範囲と、関連トピックの参照先**を引用ブロックで書く。
- 例：

```markdown
> **このドキュメントの守備範囲**：システム全体の論理構造、コンポーネントの責務、データ・ジョブの流れ。
> **使うフレームワーク・ライブラリ・サービスの具体名や選定理由**は [07_tech_stack.md](./07_tech_stack.md) を参照。
```

## 5. 編集時のチェック

要件定義書を編集する際は以下を確認する：

- [ ] 追記する内容が他ファイルと重複していないか
- [ ] 重複が発生した場合、片方を「正」として他方をリンクに置き換えたか
- [ ] 各セクション末に必要な相互リンクが貼られているか
- [ ] ファイルごとの守備範囲（上記表）から外れた内容を書いていないか

## 6. 例外

- **概要レベルの一文の引用**（例：「Backend API は NestJS で実装」）は、そのファイルの文脈上必要なら重複可。ただし詳細説明は単一箇所に集約する。
- **概念図・全体構成図** に技術名が登場するのは可（図は概観の理解を優先）。
