# ログインのしくみ（エンジニアじゃない人向け）

このサイトは「GitHub アカウントでログイン」だけを使います。
パスワードを別に作る必要はありません。

ここでは、ボタンを押してから「ログインできた！」になるまで、
裏で何が起きているかを **絵と例え話だけ** で説明します。

---

## 登場人物

```
  ブラウザ    サイト         Redis           Postgres         GitHub
  (あなた)    (このサイト)    (裏方のメモ帳)   (裏方の本棚)     (本人確認所)
   👤         🏪             📋              📚               🏛️
```

- **ブラウザ**：あなたが使っているもの
- **サイト**：問題を出してくれる学習サービス
- **Redis**：サイトの奥にある、超高速で書き込み・読み出しができる裏方のメモ帳。
  「1 回使い切りの state トークン」「ログイン中ユーザーのセッション」「レート制限のカウンタ」など、
  **一時的に覚えておけばいい情報** を全部ここに書く
- **Postgres**：サイトの奥にある、しっかりした裏方の本棚（データベース）。
  「あなたのアカウント情報」「問題」「学習履歴」など、**絶対に消えたら困る情報** を
  整理して長く保存する。Redis より遅いが、その分とても丁寧に書く
- **GitHub**：あなたが普段から使っている、世界中のプログラマが集まる場所。
  ここに「この人は本物の○○さんですよ」と保証してもらいます

### Redis と Postgres は別物

```
        Redis                            Postgres
        📋                                📚
   ┌──────────────────┐             ┌──────────────────┐
   │ 一時的なもの専用   │             │ 長く保存するもの  │
   │ (個人情報なし)    │             │ (個人情報あり)    │
   │                  │             │                  │
   │  ・state トークン │             │  ・あなたの情報   │
   │  ・セッション ID  │             │    (名前、メール) │
   │  ・内部 UUID     │             │  ・問題           │
   │  ・レート制限     │             │  ・解答・採点結果 │
   │    カウンタ      │             │  ・学習履歴       │
   │                  │             │                  │
   │ 期限が来たら自動  │             │ 消さない限り      │
   │ で消える         │             │ 残り続ける(丁寧)  │
   │ (超高速)         │             │                  │
   └──────────────────┘             └──────────────────┘
```

なぜ分けるかというと、**性質がまったく違う情報** だからです：

- **Postgres**：あなたのアカウント情報や学習履歴は **絶対に消えたら困る**
  情報。停電や再起動があっても消えないように、しっかり保管されます。
  代わりに、書き込みや検索にはちょっと時間がかかります
- **Redis**：state トークンやセッションは **消えてもまた作り直せる** 情報。
  代わりに **超高速（インメモリ KVS）** でやり取りできる必要があります
  （毎リクエストでセッション照合が走るため、ここが遅いとサイト全体が遅くなる）

> 🔒 **Redis に個人情報は入れていません**
>
> 名前 / メールアドレス / GitHub の ID といった **個人情報は Postgres にだけ保存** し、
> Redis には「内部 UUID」「セッション ID（ランダム値）」「state トークン」「IP 別の
> レート制限カウンタ」など、**それ単体では誰のものか分からない値** しか書きません。
> 万が一 Redis のデータが見えても、Postgres と突き合わせない限り個人を特定できない
> 設計にしてあります。退会時の個人情報削除も Postgres 側だけで完結します。

### この 2 つの使い分けの例え話

ホテルのフロントを想像してください：

- **Postgres** ＝ **宿泊台帳**。
  全ての宿泊客の情報を正式に記録する大きな帳簿。停電しても残る
- **Redis** ＝ フロントの机の上にある **「現在チェックイン中の客」の付箋**。
  すぐに参照できる位置に貼ってあり、チェックアウトしたらサッと剥がす

> 宿泊台帳には全宿泊客が載っていますが、今ロビーにいる客が誰かを
> 知りたいときに毎回ぶ厚い台帳をめくるのは大変。代わりに机の付箋を見れば
> 一瞬で分かる、という分担です。

## なぜ GitHub に保証してもらうの？

サイトは **「あなたのパスワードを預からない」** 方針です。

理由：
- パスワードを預かると、漏れたら大事故になる
- そもそも GitHub のアカウントを持っている人向けのサイトなので、
  わざわざ新しいパスワードを作ってもらう必要がない
- 「本人確認」はプロ（GitHub）に任せて、サイトは
  「学習機能を提供する」ことに集中したい

例えるなら、コンビニで身分証明を求められたとき、
**運転免許証（GitHub）を見せれば済む** ようなものです。
コンビニ自身があなたの顔写真を新しく登録する必要はありません。

- **Redis** が消えても、**Postgres** にアカウントは残っているので、もう一度ログインすれば
  すぐ復帰できる（このサイトではセッションだけ作り直されます）

## もう少し詳しく：6 つのステップ

### ステップ 1：ログインボタンを押す

```
ブラウザ ──「GitHub でログイン」ボタン押下──► サイト ──書き込み──► Redis
                                              🏪                  📋
                                                                  「state トークン: ABC123」
                                                                  「期限: あと 10 分」
```

サイトはこのときに **「state トークン」** をひとつ作って、**Redis に書き込み** ます。
この state トークンは **TTL（自動失効時間）10 分、1 回使い切り** のものです。

Redis には「**この state トークンは 10 分で自動的に消してね**」という TTL も
一緒に書きます。サイトが後で消し忘れても、Redis が TTL に従って自動で消してくれます。

> 🤔 なぜ state トークン？
> OAuth コールバックで GitHub から戻ってきたリクエストが「本当にさっき自分が
> 開始したものか」を確認するためです。**CSRF（クロスサイトリクエスト
> フォージェリ）対策** の標準的な方法で、悪い人が偽のコールバック URL で
> 別人のアカウントにログインさせる攻撃を防ぎます。

> 🤔 なぜ Redis に書くの？
> サイト自身が覚えていてもいいのですが、サイトの店員（プロセス）が複数いる場合、
> 「ボタンを押したときに対応した店員」と「GitHub から戻ってきたときに
> 対応する店員」が違うことがあります。みんなが見られる共通の Redis に
> 書いておけば、どの店員でもstate トークンを確認できます。

---

### ステップ 2：GitHub に連れて行かれる

```
ブラウザ ──連れて行かれる──► GitHub の画面
                              「○○さん、サイトへのログインを許可しますか?」
```

GitHub の画面で **「許可する」「キャンセル」** を選びます。
ここで初めて GitHub のパスワード（または GitHub 側のログイン状態）が
関わります。**サイトはあなたのパスワードを見ません**。

---

### ステップ 3：GitHub が「本人ですよ」と保証

```
GitHub ──「この人は○○さん本人。認可コードをお渡しします」──► サイトに戻る
```

「許可する」を押すと、GitHub が **「○○さん本人を確認しました」** という
**認可コード（code パラメータ）** を持たせて、ブラウザをサイトに送り返します
（このコードを後でサイトが GitHub に提示すると、アクセストークンに交換できる）。

---

### ステップ 4：サイトに戻ってきたところでstate トークンチェック

```
サイト ：「state トークンは?」
ブラウザ：「これです」（GitHub が一緒に持たせてくれたstate トークン）
            │
            ▼
       ┌──────┐
       │Redis │ ──「ABC123? あります! 一致!」
       │ 📋   │     ＋ その場で消す
       └──────┘
            │
            ▼
       一致 ──► OK!
            │
            └► 一致しなければ → 入り口に戻す（ログイン失敗）
```

ここで、ステップ 1 で **Redis に書き込んだstate トークン** が登場します。
サイトは **Redis に「ABC123 ありますか?」と問い合わせ**、見つかれば「本人確認」と
「悪い人にハイジャックされてないこと」の両方が確認できたことになります。

state トークンは **一致したらすぐに Redis から消します**（1 回しか使えない）。
仮に同じstate トークンで 2 回目のアクセスが来ても、もう Redis にないので拒否されます。

---

### ステップ 5：サイトが「セッション」を渡す（Postgres にも記録）

```
サイト ──セッションを発行──► ブラウザ
 🏪                       👤
 │
 │ ①初めての人なら Postgres に登録／既に居れば最新情報に更新
 ▼
┌────────┐
│Postgres│ 「○○さん、GitHub の id、名前、メール」
│ 📚     │
└────────┘
 │
 │ ② Redis に「セッション → ○○さん」の対応を書く
 ▼
┌──────┐
│Redis │ 「セッション #A7B3 → ○○さん」
│ 📋   │  「期限: 7 日」
└──────┘
```

ここでサイトは **2 つのこと** を同時にやります：

**① Postgres への記録**
- **初めてのログイン** なら、Postgres の users テーブルにアカウントを新規 INSERT します
  （GitHub の ID・名前・メールアドレスを記録）
- **2 回目以降** なら、users 行は既にあるので、GitHub 側で名前を変えていたら
  Postgres の name / email を UPDATE で最新に上書きします（再ログインのたびに自動同期）

**② Redis への記録**
- Redis に「セッション #A7B3 = 利用者番号 550e...」の対応を書きます
- このメモは **7 日で Redis 側が自動で消す** ようにセット

> 🔒 **Redis に書くのは「番号」だけ。名前やメールは書きません**
>
> 図では分かりやすさのため「○○さん」と書いていますが、実際に Redis に
> 入れているのは **名前ではなく利用者番号（user_id という内部の UUID）** です。
> 名前・メールは Postgres にしかなく、この番号を Postgres と突き合わせて
> 初めて「○○さん」と分かる仕組みです（→ 後述の 🔒 ボックスにコードベースの中身を掲載）。

そしてブラウザには **セッション ID（ランダムな番号だけ）** を `session_id` という
名前の Cookie で渡します。ブラウザに渡るのは「番号」だけで、「誰なのか」の
中身は Redis 側にしか存在しません。

```
       ブラウザ                            Redis
   ┌────────────┐                  ┌──────────────────────────┐
   │ セッション:    │ ─照合用に問合せ─► │ session:A7B3 → {         │
   │  #A7B3     │                  │   user_id: 550e...       │
   └────────────┘                  │   (= 利用者番号。名前ではない) │
                                    │   csrf_token: XYZ        │
                                    │   最後にアクセス: 19:30   │
                                    │   期限: 7 日             │
                                    │ }                        │
                                    └──────────────────────────┘
```

> 🔒 **Redis に保存している中身（コードベースの実態）**
>
> 上の図は簡略版です。実際に Redis に書いている内容は、コードを見ると次の通りです
> （実装は `apps/api/app/core/session.py`）。**どれも「それ単体では誰のものか
> 分からない値」だけ** で、名前・メールアドレスといった個人情報は一切入れていません。
>
> | 保存先（Redis のキー） | 中身 | 役割 |
> |---|---|---|
> | `session:<セッションID>` | `user_id`（利用者番号 = Postgres の users.id と同じ UUID） | このセッションが誰のものか（番号で） |
> | 〃 | `csrf_token` | 仕掛け 2 の照合用トークン |
> | 〃 | `created_at` / `last_seen_at` | 作成時刻・最終アクセス時刻 |
> | `user:<利用者番号>:sessions` | そのユーザーのセッションID 一覧 | 複数端末を一括ログアウトするため |
>
> - **個人情報（名前・メール・GitHub の ID）は Postgres にだけ** 保存し、Redis には
>   置きません（→ 冒頭の「Redis と Postgres は別物」の 🔒 ボックスと同じ方針）
> - `user_id` は `550e8400-e29b-...` のような内部番号なので、万が一 Redis が
>   見えても、Postgres と突き合わせない限り個人を特定できません
> - 退会時の個人情報削除も Postgres 側だけで完結します

セッションは **ブラウザの中に保存** され、次に何かするたびに自動で持参されます。
サイトは受け取ったセッションを Redis に問い合わせて「○○さんね、どうぞ」と
本人を識別します（あなたが意識する必要はありません）。

> 🤔 なぜセッション情報をすべて Cookie に詰めないの？
> Cookie にユーザー情報を直接入れる方式（JWT 等の self-contained token）もありますが、
> このサイトは **「Cookie には ID だけ、本体は Redis」** の **サーバーサイドセッション方式** を
> 採用しています。理由は 2 つ：
> - **失効を即時にできる**：問題が起きたら Redis から該当エントリを DELETE するだけで
>   そのセッションを即座に無効化できる（JWT は失効までトークンが有効なまま）
> - **Cookie が漏れてもサイト側で握りつぶせる**：番号だけなら情報量がなく、
>   サイト側の状態管理でコントロールできる範囲が広い

### ステップ 6：ログイン完了、学習開始！

```
ブラウザ ──「問題を見せて」+ セッション #A7B3──► サイト
                                              🏪
                                              │
                          ① 誰のセッション?        │
                                              ▼
                                       ┌──────┐
                                       │Redis │ ──「#A7B3 は○○さん」
                                       │ 📋   │
                                       └──────┘
                                              │
                          ② 問題を取り出す     │
                                              ▼
                                     ┌────────┐
                                     │Postgres│ ──「問題リストはこちら」
                                     │ 📚     │
                                     └────────┘
                                              │
                                              ▼
                                      ○○さん用の問題を返す
```

これ以降、ブラウザを閉じたり、別のページに行ったりしても、
セッションが残っている間は **再ログイン不要** で使えます。

裏では **ページを開くたびに** サイトがこの 2 ステップを実行します：

1. **Redis** に「このセッションは誰?」を問い合わせる（超高速、本人確認）
2. **Postgres** から「その人に表示すべき情報」を取り出す（少し時間かかるが正確）

役割分担で、**「速さ」と「確実な保管」の両方** を実現しています。

## ログイン後にページアクセスしたときの裏側の流れ

ログイン後、ブラウザでページを移動したり何かのボタンを押したりするたびに、
裏で同じ流れが繰り返されます。ここで詳しく分解します。

### パターン A：問題を読みに来た（GET リクエスト = 情報を取り出すだけ）

例：`/problems` を開く、`/me/stats` を開く、ヘッダーで自分の名前を表示する

```
┌──────────┐        GET /problems                    ┌──────────┐
│ ブラウザ  │ ───────────────────────────────────►   │ サイト    │
│  👤      │  Cookie: session_id=signed.A7B3xyz...   │  🏪       │
└──────────┘                                          └────┬─────┘
                                                           │
                                       ┌───────────────────┘
                                       │
              ① Cookie の電子署名を確認 │
                                       ▼
                            ┌────────────────────┐
                            │ HMAC 署名検証       │
                            │ "signed.A7B3xyz..."│
                            │ → 改ざんなし ⭕    │
                            │ → セッション ID:    │
                            │   "A7B3"           │
                            └────────┬───────────┘
                                     │
              ② Redis に session:A7B3 を問い合わせ
                                     ▼
                            ┌────────────────────┐
                            │ Redis              │
                            │ 📋                  │
                            │ session:A7B3 →     │
                            │   user_id=42       │
                            │   csrf_token=XYZ   │
                            │   last_seen=14:30  │
                            │   期限: あと 6 日 │
                            └────────┬───────────┘
                                     │
                                     │ Redis ヒット ⭕
                                     │ user_id = 42
                                     │ (期限切れなら 401 で
                                     │  Cookie 自動削除)
                                     ▼
              ③ Postgres から user_id=42 を取得
                            ┌────────────────────┐
                            │ Postgres           │
                            │ 📚                  │
                            │ SELECT * FROM users│
                            │ WHERE id = 42      │
                            │   AND deleted_at   │
                            │   IS NULL          │
                            │ → 山田太郎の情報   │
                            └────────┬───────────┘
                                     │
                                     │ Postgres ヒット ⭕
                                     │ (deleted_at が
                                     │  入っていれば 401)
                                     ▼
              ④ 業務処理 (山田さんに表示すべき問題を取得)
                            ┌────────────────────┐
                            │ Postgres           │
                            │ 📚                  │
                            │ SELECT * FROM      │
                            │ problems ...       │
                            │ → 問題リスト       │
                            └────────┬───────────┘
                                     │
                                     ▼
              ⑤ レスポンス (HTML / JSON) を返す
                                     │
                                     ▼
┌──────────┐        200 OK                            ┌──────────┐
│ ブラウザ  │ ◄───────────────────────────────────   │ サイト    │
│  👤      │   問題リストの JSON                       │  🏪       │
└──────────┘                                          └──────────┘
              画面に「山田さん」「問題リスト」を表示
```

**所要時間の目安**：

| ステップ | 時間 |
|---|---|
| ① 電子署名検証 | 数 μs（CPU だけ） |
| ② Redis 問い合わせ | 0.1〜1 ms |
| ③ Postgres ユーザー取得 | 1〜3 ms |
| ④ 業務処理 | 数 ms〜数十 ms |
| **認証オーバーヘッド合計** | **2〜5 ms 程度** |

普通の API は数十〜数百 ms かかるので、認証で 2〜5 ms 足されても **体感できないレベル** です。

---

### パターン B：操作を確定した（POST / PUT / DELETE = 状態を変える）

例：解答を送信する、ログアウトする、設定を更新する

GET のときの 5 ステップに加え、**CSRF トークンの照合** が間に挟まります。

```
┌──────────┐  POST /submissions                       ┌──────────┐
│ ブラウザ  │ ───────────────────────────────────►   │ サイト    │
│  👤      │  Cookie: session_id=signed.A7B3xyz...    │  🏪       │
│          │          csrf_token=XYZ                  │          │
│          │  Header: X-CSRF-Token: XYZ ❗             │          │
│          │  Body:   解答コード                       │          │
└──────────┘                                          └────┬─────┘
                                                           │
              ① Cookie の電子署名を確認 (パターン A と同じ)
              ② Redis に session を問い合わせ (同上)
                                                           │
              ❗ CSRF トークン照合 (POST/PUT/DELETE のみ)   │
                                                           ▼
                            ┌────────────────────┐
                            │ Cookie の          │
                            │  csrf_token: XYZ   │
                            │                    │
                            │ ヘッダの           │
                            │  X-CSRF-Token: XYZ │
                            │                    │
                            │ → 一致 ⭕         │
                            └────────┬───────────┘
                                     │
                                     │ 不一致なら 403 Forbidden
                                     │
              ③ Postgres から user 取得 (パターン A と同じ)
              ④ 業務処理 (解答を保存、採点ジョブを投入)
              ⑤ レスポンスを返す
                                     │
                                     ▼
┌──────────┐        202 Accepted                      ┌──────────┐
│ ブラウザ  │ ◄───────────────────────────────────   │ サイト    │
│  👤      │   submissionId を返す                    │  🏪       │
└──────────┘                                          └──────────┘
```

**CSRF トークン照合の意味**：

ポイントは「**Cookie は自動で付く / ヘッダは自分で読んで付ける**」という
2 つの経路の性質の違いです。

```
[本物の画面からの POST] サイト自身の画面が送る
  Cookie:  csrf_token=XYZ    ← ブラウザが自動で付ける
  Header:  X-CSRF-Token=XYZ  ← サイトの JS が値を読んで乗せた
  → 照合 XYZ == XYZ → 一致 ⭕ 実行

[罠サイトからの偽 POST] 別ドメインの罠ページが送る
  Cookie:  csrf_token=XYZ    ← ブラウザが勝手に付けてしまう（防げない）
  Header:  X-CSRF-Token=???  ← ここに XYZ を入れられない！
  → 照合 ??? != XYZ → 不一致 ❌ 403 で弾かれる
```

なぜ攻撃者は正しい値をヘッダに入れられないのか：
- Cookie は **送り先ドメインを見て自動で付く** ので、罠サイトからでも運ばせられる
- でもヘッダに合言葉を乗せるには、その値を **読み取る** 必要がある
- 罠サイトは **別ドメイン** なので、本物サイトの Cookie も JS が持つ値も
  読めない（ブラウザの同一オリジン制約）→ 正しい値を入れられない

つまり「Cookie は運ばせられても、合言葉は読めない」という非対称性で、
**本物の画面から出た POST と罠サイトから出た POST を見分けています**。
これが「**Cookie 自動付与 + 状態変更 API**」を CSRF から守る仕組みです。

---

### パターン C：30 分以上アクセスがなかった (期限の自動延長)

最後にアクセスしてから 30 分以上経つと、Redis のセッション期限を
**「あと 7 日」にリセット** します。利用者には見えませんが、
**使っている間は切れない** の正体はこれです。

```
[最終アクセス 14:30、現在 15:05 のリクエスト]

ブラウザ ──GET /problems──► サイト
                              │
                              │ Redis から session を取得
                              │  → 最終アクセスは 14:30
                              │  → 現在は 15:05
                              │  → 35 分経過 (>= 30 分)
                              │
                              ▼ TTL リセット ❗
                       ┌────────────────────┐
                       │ Redis              │
                       │ 📋                  │
                       │ session:A7B3       │
                       │   last_seen=15:05  │ ← 更新
                       │   期限を 7 日に    │ ← 延長
                       │   再設定          │
                       └────────┬───────────┘
                                │
                                │ 以降は通常の処理
                                ▼
                          業務処理を続行


[最終アクセスから 30 分以内なら何もしない]

ブラウザ ──GET /problems──► サイト
                              │
                              │ Redis から session を取得
                              │  → 35 分経過 (< 30 分なら)
                              │
                              ▼ TTL リセットしない
                          そのまま業務処理へ
```

**なぜ 30 分間引き?**：毎リクエストで EXPIRE すると Redis 負荷が線形に増えるため、
**「最終アクセスから 30 分以上経った時だけ延長」** という間引きで負荷を抑えています。
ユーザー体験は「7 日放置でログアウト」と等価で、違和感はありません。

---

### パターン D：セッション期限切れだった

7 日間操作がなく、Redis がセッションを自動削除した後に来た場合。

```
ブラウザ ──GET /problems──► サイト
   Cookie: session_id=signed.A7B3xyz...
                              │
                              │ Cookie の電子署名は OK
                              │ Redis に session:A7B3 を問い合わせ
                              │
                              ▼
                       ┌────────────────────┐
                       │ Redis              │
                       │ 📋                  │
                       │ session:A7B3       │
                       │   → 存在しない ❌  │
                       │   (TTL で自動削除済)│
                       └────────┬───────────┘
                                │
                                ▼
              401 Unauthorized + Set-Cookie で Cookie 削除
                                │
                                ▼
ブラウザ ◄────401 + Set-Cookie: session_id=; Max-Age=0─── サイト
   Cookie 削除済
                                │
                                ▼
              ブラウザは `/login` に遷移して再ログインを促す
              (期限切れ Cookie が残らないので無限ループしない)
```

これが **仕掛け 4：期限切れセッション Cookie の自動削除** の実際の動きです。

---

### 一目で見る「毎リクエストの処理マトリクス」

```
                     GET (読み取り)    POST/PUT/DELETE (書き込み)
                     ──────────────    ─────────────────────────
   ① 電子署名検証    ⭕                ⭕
   ② Redis 引き     ⭕                ⭕
   ③ CSRF 照合      —                 ⭕ ❗
   ④ Postgres 引き  ⭕                ⭕
   ⑤ TTL 延長判定   ⭕ (30 分以上なら) ⭕ (同上)
   ⑥ 業務処理       ⭕                ⭕
```

すべてのリクエストで通る共通処理が認証の土台で、
書き込み系だけ CSRF 照合という 1 層が追加で挟まる、というシンプルな構造です。

## ログイン後の便利な仕組み

### 自動延長（使ってる間は切れない）

```
最終アクセスから 7 日間操作なし → Redis が自動でそのセッションを消す
                                       │
                                       ▼
                                  セッションを持ってきても
                                  「ありません」と返る
                                       │
                                       ▼
                                  ログイン画面に戻される
```

Redis には **「7 日間使われなかったら自動で消してね」** という設定で
書かれています。利用者が操作するたびにこの期限を「あと 7 日」に
リセットするので、使っている間は切れません。

例えるなら、ジムの会員カードのようなもの。
**使っているうちは期限を気にしなくていい** けれど、
1 週間まったく来なかったら念のため再入会手続きをしてもらう、という運用です。

```
月  火  水  木  金  土  日  月  火 ...
🟢  🟢  🟢  ⬛  ⬛  ⬛  ⬛  🟢 ← 操作したのでタイマーが「あと 7 日」に戻る

🟢 = 操作した日（タイマーリセット）
⬛ = 操作なし（タイマー減る）
```

### ログアウト

```
ブラウザ ──「ログアウト」ボタン押下──► サイト
                                         🏪
                                         │
                                         ▼
                                    ┌──────┐
                                    │Redis │ ──「#A7B3 を消去」
                                    │ 📋   │
                                    └──────┘
                                         │
                                         ▼
                                    ブラウザのセッションも削除
                                         │
                                         ▼
                                    ホーム画面に戻る
```

ログアウトすると、**Redis からセッションの対応を消し、ブラウザのセッションも削除** します。
共有パソコンを使った後などは、**念のためログアウトしておくと安全** です。

> 🤔 Postgres のアカウント情報はどうなるの？
> **Postgres はそのまま残ります**。ログアウトは「セッションだけを無効化する」操作で、
> アカウント自体を削除するわけではありません。次にログインすれば
> 同じアカウントで学習履歴も引き継がれます（アカウント削除＝退会は別の機能）。

## 安全のための見えない仕組み

裏ではあなたを守るためにいくつかの仕掛けがあります。
全部は説明しませんが、雰囲気だけ。

### 🛡 仕掛け 1：1 回使い切りの state トークン（ステップ 1〜4 で説明済み）

悪い人が「ここをクリック！」と偽のログインリンクを送りつけても、
state トークンが合わないので **ログインに成功しない** 仕組みです。

### 🛡 仕掛け 2：操作の度に「本人ですか?」のCSRF トークン

ログイン後に POST / PUT / DELETE / PATCH（**状態を変える HTTP メソッド**）で
リクエストするときは、`session_id` Cookie に加えて **`X-CSRF-Token` ヘッダ** に
CSRF トークンを乗せる必要があります（double submit cookie 方式）。

```
ブラウザ ──「POST /auth/logout」+「X-CSRF-Token: XYZ」──► サイト
                                                          │
                                                          ▼
                                                  Cookie の値とヘッダの値を照合
                                                  一致 → 実行
                                                  不一致 → 403 Forbidden
```

これは **CSRF（クロスサイトリクエストフォージェリ）攻撃** への防御です。
他サイトに仕掛けられた罠フォームから勝手にこのサイトの状態変更 API を
叩かれても、攻撃者は Cookie の値を読み取ってヘッダに乗せることができない
（同一オリジン制約）ため、リクエストが防がれます。

### 🛡 仕掛け 3：レート制限（Rate Limit）

`/auth/github` 系のエンドポイントを **同一 IP から 1 分間に 10 回以上** 叩くと、
サイトはレート制限で **HTTP 429（Too Many Requests）** を返してブロックします。

```
1 回目  ✅ 200 / 302
2 回目  ✅
...
10 回目 ✅
11 回目 ❌ 429 Too Many Requests
```

これは、悪い人が **自動プログラムでログイン経路を総当たり試行する**
（state トークンの総当たり攻撃、GitHub API への過剰呼び出し、mock ログイン経路の
悪用など）のを防ぐためです。普通に使っている人がこの上限に当たることはありません。

### 🛡 仕掛け 4：期限切れセッション Cookie の自動削除

期限切れセッションでアクセスすると、サーバは **401 Unauthorized** を返すと同時に、
**Set-Cookie ヘッダで `session_id` Cookie を Max-Age=0 で削除** するよう指示します。

```
ブラウザ ──「期限切れ session_id」──► サイト：「401 + Set-Cookie で削除」
                                          │
                                          └─► ブラウザの session_id Cookie が消える
                                               │
                                               └─► 次のアクセスは未認証として扱われる
```

これがないと、期限切れ Cookie を持ったまま `(authed)` レイアウトと `/login` の間で
**無限リダイレクトループ** に陥る可能性があるため、サーバ側で確実に Cookie を
片付ける契約になっています。

### 🛡 仕掛け 5：Cookie 属性で「JS から読めない / HTTPS でしか飛ばない」

セッション Cookie には 3 つの安全属性が付いています。

| 属性 | 何をする |
|---|---|
| `HttpOnly` | **JavaScript から `document.cookie` で読めなくなる**（XSS で盗まれない）|
| `Secure` | **HTTPS の通信でしかブラウザがサーバに送らない**（盗聴対策、本番のみ） |
| `SameSite=Lax` | **他サイトからの遷移リクエストには付いて行かない**（CSRF の追加防御）|

```
攻撃者が罠スクリプトで盗もうとしても…
  <script>fetch("/", { cookies }) → ❌ HttpOnly で読めない
  http:// で通信を傍受しようとしても → ❌ Secure で平文では飛ばない
  他サイトから fetch されても       → ❌ SameSite で付いて行かない
```

普通の利用者には見えない、**ブラウザに任せる構造的な防御** です。

### 🛡 仕掛け 6：再ログイン時の旧セッション自動廃棄

ログイン処理の最初に **「ブラウザに残っている古いセッションがあれば Redis から削除」**
してから新しいセッションを発行します。

```
（再ログイン時）
ブラウザ ──旧 session_id 付き──► サイト
                                  │
                                  ├─► Redis から旧セッションを DELETE
                                  │
                                  └─► 新しい session_id を発行して上書き
```

これは **共有 PC や Cookie が漏れた場合の防御** です。誰かが旧セッション ID を
握ったまま使えてしまうと困るので、再ログインのタイミングで強制的に無効化します。

### 🛡 仕掛け 7：オープンリダイレクト対策

`/login?next=...` の戻り先 URL を **サイト内のパスだけに制限** し、外部 URL や
偽装パスは弾いてホーム `/` に倒します。

```
✅ /login?next=/problems              → ログイン後 /problems に遷移
❌ /login?next=https://evil.com        → 外部 URL なのでホームに
❌ /login?next=//evil.com              → スキーム省略の罠も弾く
❌ /login?next=/%2F%2Fevil.com         → URL エンコード経由の罠も弾く
❌ /login?next=<2048 字超>             → 異常に長い値も弾く
```

これは **フィッシング対策** です。攻撃者が「`https://正規サイト/login?next=偽サイト`」
というリンクを送りつけて、ログイン直後に偽サイトへ誘導するのを防ぎます。
URL の **見た目は本物のドメイン** なので、利用者が気づきにくいタイプの攻撃です。

### 🛡 仕掛け 8：Cookie の改ざん検知（電子署名）

`session_id` Cookie の値には **サイトしか作れない電子署名（HMAC）** が付いていて、
ブラウザ側で値をいじっても **「これは偽物」** と即座に分かるようにしてあります。

```
本物の Cookie:  "A7B3xyz.SIGNATURE_a3f9b2..."
                  └─値─┘ └────署名────┘

攻撃者が中身を書き換える:
                "OTHER.SIGNATURE_a3f9b2..."  ← 署名は古いまま
                                                │
                                                ▼
                                          サーバ側で署名検証 → 失敗 → 401
```

ブラウザは Cookie を自由に書き換えられますが、署名鍵はサーバだけが知っているので、
**攻撃者は正しい署名を作れません**。他人のセッション ID を推測して紛れ込ませる
攻撃を構造的に防ぎます。

## 採用しなかった方式

設計時に検討したが採用しなかった代表的な方式を簡潔にまとめます。
「なぜこの設計なのか」を理解する補助として残しておきます。

### ① JWT（自己完結トークン）

トークンの中にユーザー情報を埋め込み、サーバ側で保管しない方式。

> ℹ️ **JWT の「運び方」について**
> JWT は **トークンの中身のフォーマット** であって、運び方ではありません。
> 一般的には `Authorization: Bearer <jwt>` ヘッダで運ぶ例（API / モバイル）が
> 多いですが、**Cookie に入れて運ぶ** こともできます。ここでの比較は
> **「JWT を HttpOnly Cookie で運ぶ版」** を想定（XSS 耐性を session_id と
> 揃えるため）。Bearer ヘッダ + localStorage 版は **XSS でトークン丸ごと
> 盗まれるリスク** が追加で乗るので、Web アプリでは更に不利になります。

| 利点 | 欠点（採用しなかった理由） |
|---|---|
| Redis も Postgres も毎回引かなくて良い | **即時失効ができない**（トークン有効期限まで使える）|
| サーバ側の状態管理が不要 | Cookie に個人情報相当が乗る |
| | 失効や名前変更の追随が複雑 |

→ **「即時失効」と「個人情報を Cookie に載せない」を優先** してサーバーサイドセッションを採用。

> 🔑 **「即時失効」の中身**
> このプロジェクトは毎リクエストで Redis + Postgres を引くので、
> **Redis から該当セッションを DELETE** すれば次のリクエストから即 401。
> さらに **Postgres の `users.deleted_at` をセット** するだけでも、Repository 層の
> `WHERE deleted_at IS NULL` フィルタによって自動的に 401 になる二重防御。
> JWT 方式だとどちらの操作も次回トークン検証時には反映されず、有効期限が
> 切れるまで使えてしまうのが致命的な差。

### ② Redis にユーザー情報もキャッシュ

毎リクエストの Postgres 問い合わせを減らすため、名前・メールも Redis に置く方式。

| 利点 | 欠点（採用しなかった理由） |
|---|---|
| Postgres アクセスが減る | **Redis に個人情報が乗る**（この設計の方針に反する）|
| | キャッシュ無効化が必要（二重管理）|
| | 退会・名前変更後も古い値が残る可能性 |

→ **「個人情報は Postgres 1 か所だけ」を維持**。主キー検索は十分速い（1〜3 ms）ので最適化不要。

### ③ state トークンを Cookie に保存

OAuth の state を Redis ではなくブラウザの Cookie に保存する方式。

| 利点 | 欠点（採用しなかった理由） |
|---|---|
| Redis 不要で簡単 | **複数タブで上書きされて壊れる**（後発が先発を潰す）|
| | 1 回使い切りをサーバ側で強制できない（ブラウザ任せ）|
| | 改ざん検知のため署名鍵管理が必要 |
| | next（戻り先 URL）と Cookie が分散する |

→ **マルチタブ耐性と 1 回使い切りの構造的保証** を優先して Redis に保存。

### ④ ユーザー情報をブラウザに永続保存（localStorage 等）

`/auth/me` で取った名前・メールをブラウザ側でずっと保持しておく方式。

| 利点 | 欠点（採用しなかった理由） |
|---|---|
| サーバへの問い合わせ回数が減る | **個人情報がブラウザのディスクに残る**（共有 PC 等でリスク）|
| | GitHub 側で名前変更しても追随しない |
| | XSS で全部読まれる |

→ **タブを閉じたら消える「メモリキャッシュ（TanStack Query）」だけを採用**。
   個人情報をディスクに書かない設計に揃える。

### ⑤ JWT + Redis ブラックリスト方式（ハイブリッド）

純粋 JWT の「即時失効できない」弱点を補強する **現実の運用パターン**。
JWT を主体としつつ、サーバ側で **「このユーザーは時刻 T 以降のトークンだけ有効」**
を Redis に書いて、古い JWT を一斉無効化する。

#### 仕組み

```
[通常の API リクエスト]
  ブラウザ ──JWT──► サーバ
                     │
                     │ ① JWT 署名検証（秘密鍵で再計算）
                     │
                     │ ② Redis に「ブロック時刻」を問い合わせ ❗
                     │    キー: "blacklist:user:<userId>"
                     │    値: ブロックされた時刻（unix 秒）
                     │
                     │    if (JWT の iat < ブロック時刻)
                     │      → そのユーザーの旧 JWT は無効
                     │    else
                     │      → 通す
                     ▼
                  応答


[パスワード変更 / 退会 / BAN 時]
  サーバ:
    ① Redis に「ユーザー X はこの時刻以降の JWT だけ有効」と書く
    ② 必要なら新しい JWT を即発行（iat はブロック時刻以降）
       ↓
  攻撃者が手元に持っていた旧 JWT (iat < ブロック時刻) → 即拒否
  本人の新 JWT (iat > ブロック時刻)                  → 通る
```

ポイント：**JWT 1 個ずつではなく「時刻線」で管理** するので、Redis に書くのは
ユーザー 1 人につき 1 行で済む（個別 JWT のブラックリストより効率的）。

| 利点 | 欠点（採用しなかった理由） |
|---|---|
| **即時失効ができる**（純粋 JWT の弱点を解消） | 毎リクエストで Redis 引くので「完全ステートレス」のメリットは部分的に失う |
| モバイル / マイクロサービス展開がしやすい | Cookie に乗せない場合 XSS でトークンを盗まれるリスク |
| Bearer ヘッダ前提の OAuth エコシステムと相性◎ | ペイロードに個人情報が乗りうる |

→ **このプロジェクトでは Web 完結＋シンプルさを優先** したため、最初から
   不透明 ID + Cookie を選択。ただし **JWT を採用する現実のプロジェクトの多くは
   実はこの方式** で運用しており、「JWT は即時失効できない」は純粋ステートレス時の
   話であって、実務では補強パターンが定着している。

> 🔑 **「JWT 採用 = 純粋ステートレス」ではない**
> 実務でほぼ全ての JWT プロジェクトが何らかの補強策（ブラックリスト・iat 比較・
> 短命トークン + リフレッシュ等）を入れている。「JWT を選ぶ」は **「ペイロードに
> 認証情報を載せる方式を選ぶ」** と理解するのが正確で、ステートレス性は
> サービス要件に応じて部分的に犠牲にする。

### ⑥ 認証方式 5 パターンの一覧（参考）

「トークンの中身 × 運び方」の組み合わせ＋補強策で認証方式は 5 通りあります。
このプロジェクトがどこに位置するかを俯瞰できる表として残しておきます。

| # | 方式 | 即時失効 | XSS 耐性 | CSRF 耐性 | 個人情報の置き場 | サーバ負荷 | このプロジェクト |
|:-:|---|:-:|:-:|:-:|---|:-:|:-:|
| **①** | **純粋 JWT + Bearer ヘッダ**（localStorage） | ❌ | ❌ JS で読まれる | ⭕ 自動付与されないので強い | Cookie ＋ JS メモリ | ◎ 計算のみ | 不採用 |
| **②** | **純粋 JWT + Cookie** | ❌ | ⭕ HttpOnly で守れる | ❌ 別途対策必要 | Cookie | ◎ 計算のみ | 不採用 |
| **③** | **不透明 ID + Bearer ヘッダ**（localStorage） | ⭕ サーバ側で握る | ❌ JS で読まれる | ⭕ 強い | サーバ | ○ Redis 1 回 | 不採用 |
| **④** | **不透明 ID + Cookie**（session_id）| ⭕ サーバ側で握る | ⭕ HttpOnly で守れる | ❌ 別途対策必要 | サーバ | ○ Redis 1 回 | **採用** |
| **⑤** | **JWT + Redis ブラックリスト**（ハイブリッド） | ⭕ iat 比較で即無効化 | △ 運び方次第 | △ 運び方次第 | Cookie or JS メモリ | ○ Redis 1 回 | 不採用 |

> ①〜⑤ の番号は下の §⑦「5 パターンの使い分けガイド」と対応しています（同じ番号で参照されます）。

「即時失効」「XSS 耐性」「個人情報をサーバに閉じ込める」の 3 つを満たすには
**④（不透明 ID + Cookie）一択** で、その代わりに発生する CSRF 耐性の弱さは
**double submit cookie 方式（仕掛け 2）** で別途防御する、という設計です。

**JWT + ブラックリスト方式** はモバイル / マイクロサービス展開を見据えるプロジェクトでよく選ばれる
中間解で、即時失効を諦めずに JWT のエコシステム（OAuth・Bearer ヘッダ前提のライブラリ）を
取り込めるのが利点です。

> 用語整理：
> - **JWT**：トークンの **中身のフォーマット**（`header.payload.signature` の base64 文字列）
> - **不透明 ID**：中身に意味を持たない単なるランダム文字列。サーバ側のストア（Redis）と突き合わせて初めて意味を持つ
> - **Bearer ヘッダ**：`Authorization: Bearer <文字列>` という HTTP ヘッダで運ぶ方式
> - **Cookie**：ブラウザが自動でリクエストに付与する保存域。`HttpOnly` 属性で JS から隠せる

### ⑦ 5 パターンの使い分けガイド

「正解は 1 つ」ではなく、**サービス形態と優先するもの** で変わります。
迷ったときの判断軸を整理しておきます。

#### 早見表：サービス形態別の推奨

| サービス形態 | 推奨方式 |
|---|---|
| Web アプリ（ブラウザだけで完結） | **④ 不透明 ID + Cookie**（このプロジェクト） |
| マイクロサービス間通信 / B2B API | **① 純粋 JWT + Bearer ヘッダ** |
| モバイルアプリ（iOS / Android ネイティブ） | **① 純粋 JWT + Bearer** または **③ 不透明 ID + Bearer** |
| 公開 API（外部サービスに使わせる） | **① 純粋 JWT + Bearer ヘッダ** |
| 社内ツール / 管理画面（厳格な失効が必要） | **④ 不透明 ID + Cookie** |
| 短期セッションでログアウト概念がない | **① 純粋 JWT + Bearer ヘッダ** |
| **JWT のエコシステムを使いつつ即時失効も両立したい** | **⑤ JWT + Redis ブラックリスト** |

#### 4 つの判断軸

```
Q1. クライアントはブラウザだけ?
    Yes → Cookie 系（HttpOnly で XSS 守れる）
    No  → Bearer ヘッダ系（モバイル / API が Cookie を扱いにくい）

Q2. ログアウト・BAN を即座に効かせたい?
    Yes → 不透明 ID 系（サーバで握りつぶせる）
    No  → JWT 系（短い有効期限で代替）

Q3. 認証サーバを毎リクエスト引きたくない?
    Yes → JWT 系（自己完結で DB 不要）
    No  → 不透明 ID 系（普通のサービスなら問題なし）

Q4. クロスドメインで共通認証したい?
    Yes → Bearer ヘッダ系（Cookie のドメイン制約を回避）
    No  → Cookie 系で十分
```

#### パターン別の「これを選ぶべきとき」

**① JWT + Bearer ヘッダ**（localStorage 保管）
- モバイルアプリ、マイクロサービス間、公開 API、OAuth 2.0 アクセストークン
- 短期セッション（有効期限 15 分〜1 時間）で即時失効の必要性を回避できる
- **避ける**：ブラウザ完結の Web アプリ（XSS リスク）、即時 BAN が必要

**② JWT + Cookie**
- Web + JWT の自己完結性を取りたい（XSS は HttpOnly で抑える）
- CDN / エッジで JWT 検証するアーキテクチャ
- **避ける**：即時失効必須、マイクロサービス間

**③ 不透明 ID + Bearer ヘッダ**
- モバイル + 即時失効が両方必要（銀行アプリ、医療系）
- OAuth 2.0 の Introspection エンドポイント方式
- **避ける**：パブリック Web（XSS で盗まれる）

**④ 不透明 ID + Cookie**（このプロジェクト）
- 典型的な Web アプリ / SaaS / 社内ツール / 管理画面
- 即時失効が必須（退会・BAN・強制ログアウト）
- 個人情報をクライアントに出したくない
- **避ける**：モバイル、マイクロサービス間

**⑤ JWT + Redis ブラックリスト**（ハイブリッド、現実的な JWT 採用パターン）
- 「JWT のエコシステムを取りたいが、即時失効も諦めたくない」
- モバイル・マイクロサービス展開を見据えつつ、退会・BAN を即座に効かせたい
- 既存の Bearer 前提ライブラリ（passport-jwt 等）を活用したい
- **避ける**：完全ステートレスを死守したい大規模分散システム（Redis 単一障害点が嫌なケース）

#### ハイブリッドという選択肢

実務では **「Web は ④ Cookie、モバイル / 外部 API は ① Bearer」を併用** するのが
現実解。GitHub / Google / Microsoft など大手 SaaS はほぼこの構成です。

```
┌────────────────┐                ┌────────────────┐
│ Web ブラウザ    │ ─session_id──► │                │
└────────────────┘   Cookie       │                │
                  (= ④ 不透明 ID  │                │
                     + Cookie)    │  Backend API   │
                                  │                │
┌────────────────┐                │                │
│ モバイル / 外部 │ ─JWT────────► │                │
│ サービス        │   Bearer      └────────────────┘
└────────────────┘
              (= ① 純粋 JWT + Bearer)
```

#### このプロジェクトが ④ を選んだ理由（再掲）

| 判断軸 | 状況 | 結論 |
|---|---|---|
| クライアントは? | ブラウザだけ（モバイルなし） | Cookie OK |
| 即時失効必要? | 退会・BAN を即座に効かせたい | 不透明 ID |
| サーバ負荷は? | 小〜中規模、毎回 DB 引いて問題なし | 不透明 ID OK |
| クロスドメイン? | 単一ドメイン完結 | Cookie OK |

将来モバイルアプリを追加するなら、API 側を **「Web は Cookie、モバイルは
Bearer」のハイブリッド** に拡張するのが現実的な進化パスです。

> **判断のまとめ**
> - 「Web 完結 + 即時失効重視」なら **④**（このプロジェクト）
> - 「モバイル / API / マイクロサービス」なら **①** か **③**
> - 「Web で JWT のステートレス性を取りたい」なら **②**
> - 「JWT のエコシステムを使いつつ即時失効も両立したい」なら **⑤**
> - 大規模サービスは **①と④の両建て**、または **⑤ で統一** が現実解
>
> 迷ったら **④ から始めて、必要になった時に ① や ⑤ を追加** がリスクの少ない進化パス。
