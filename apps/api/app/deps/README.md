# deps/

## deps/ とは何か

FastAPI には `Depends(...)` という仕組みがあり、**router の引数に「自動で何かを差し込む」**ことができます。
`deps/` はその「差し込む側の関数」を集めるフォルダです。

リクエストが来るたびに FastAPI が deps の関数を実行し、戻り値を router の引数にセットしてから router 関数を呼びます。
これにより routers は「認証チェック」「DB セッション開閉」のような前処理を書かずに、業務処理だけに集中できます。

## deps/ で扱える機能パターン一覧

`Depends` は「リクエストごとに何かを準備する」用途なら何でも書ける汎用機構。
代表的なパターンを **このプロジェクトで使うもの / 使わなさそうなもの** に分けて列挙する。

### 🟢 このプロジェクトで使うパターン

#### 1. DB セッションの貸し出し（`get_async_session`）

**何のために**：「リクエスト開始時に DB セッションを開き、終了時に確実に閉じる」処理を 1 箇所に集めるため。

DB セッションは「開いたら必ず閉じる」必要があるリソースで、閉じ忘れると接続プールが枯渇してアプリ全体が止まる。
各 router で個別に開閉処理を書くと、閉じ忘れリスクが高く書き方もバラバラになる。

deps/ に集めておけば、リクエストごとに**自動で開閉**される。router は「session を受け取って使うだけ」で済むので、
リソース漏れ事故を構造的に防げる。ほぼ全 API で使う最重要 dep。

#### 2. ログイン中ユーザーの取得（`get_current_user`）

**何のために**：「未ログインのリクエストを 401 で弾く」「ログイン済みなら User オブジェクトを差し込む」を 1 箇所で完結させるため。

このプロジェクトでは認証必須エンドポイントが大半（問題取得・採点・履歴閲覧）。
各 router で個別に Cookie 検証・DB 引き当て・401 投げを書くと、認証ロジックが分散して齟齬が生まれる。

deps/ に集めれば、認証必須の router は **「current_user を受け取る」と引数に書くだけ**で済む。
未ログイン時の挙動も全 router で一貫する。将来 Google OAuth を追加する時も（[ADR 0011](../../../../docs/adr/0011-github-oauth-with-extensible-design.md)）、変更は core/ や services/ 側に閉じ、**業務 router は引数の受け取り方を変えずに済む**。

#### 3. 認可ガード（`require_admin` 等）

**何のために**：「権限が足りないユーザーを 403 で弾く」を宣言的に書き、書き忘れによる権限漏れを防ぐため。

「管理画面 API は管理者のみ」「自分の提出のみ閲覧可」のような制御は、router 関数の冒頭で `if not user.is_admin:` のような書き方も可能だが、
**書き忘れがあると権限チェックが素通り**してしまう。

deps/ にガード関数を作って `dependencies=[Depends(require_admin)]` で宣言すれば、関数本体に入る前に権限チェックされる。
書き忘れに気付きやすくなり、Swagger UI 上でも「この API は要管理者」が見える化される。

#### 4. ページネーション引数の正規化（`pagination_params`）

**何のために**：「`?page=X&limit=Y` クエリの解釈ルール」を全 list API で揃えるため。

問題一覧・提出履歴・通知一覧など、ページング付き API は複数ある。
各 router で個別に「マイナス値は 1 に丸める」「上限は 100 まで」のような正規化を書くと、書き方がブレて
**「ここの API は上限 100 だけど別の API は無制限」のような不整合**が生じる。

deps/ に集めれば、全 list API が同じルールで動く。「limit の上限を 100 → 50 に下げる」のような方針変更も 1 箇所で完結する。

#### 5. レート制限チェック（`slowapi` 統合）

**何のために**：「Bot やバグによる連打」「LLM API コストの暴走」を防ぐため。

このプロジェクトでは採点・問題生成が **LLM API を呼ぶ**ため、無制限に叩かれると 1 リクエスト数十円〜のコストが青天井で発生する。
個人開発のクレジットカードが燃えるのを防ぐ意味で必須。

deps/ にレート制限を集めれば、「LLM 系 API は 1 分 10 回」「問題取得は 1 分 100 回」のように
**API ごとの方針を宣言的に書ける**。Redis 側でカウンタを共有するので、複数の API インスタンスをまたいでも制限が効く。

#### 6. リクエスト ID の採番（`get_correlation_id`）

**何のために**：observability の trace_id 自動付与の起点を作るため。

[observability/](../observability/README.md) の「trace_id 自動付与」と連動する dep。
リクエストの最初の地点（= router 受付直後）で ID を採番しないと、後段のログ・スパンに付けようがない。

クライアント側が `X-Request-Id` ヘッダで指定してきた場合はそれを尊重し、なければ自動採番、というロジックを
deps/ に置けば、全 router で一貫した ID 採番が保証される。

### 🔴 このプロジェクトでは使わなさそうなパターン

参考までに「`Depends` で書けるけどこのプロジェクトでは出番がない」パターンも挙げておく。

#### 1. Bearer Token / API キー認証

**何のために**：外部システム連携用の API キーを `Authorization: Bearer ...` ヘッダから取り出すため。

このプロジェクトは Web UI 経由の Cookie 認証のみで、サーバー間 API（CI から叩く・他社サービスと連携する等）は無いため不要。

#### 2. CSRF Token 検証

**何のために**：悪意のある別ドメインからの POST/PUT/DELETE を防ぐため。

このプロジェクトは `SameSite=Lax` の Cookie + 同一オリジン Fetch のみで動くため、CSRF Token を別途検証しなくてもブラウザが弾く構成になる。

#### 3. ファイルアップロードの一時保存

**何のために**：`UploadFile` を受け取り、一時ディレクトリに保存するため。

このプロジェクトはコード入力テキストのみで、画像・PDF・添付ファイル機能が無いため不要。

#### 4. 多言語化（Accept-Language）

**何のために**：`Accept-Language` ヘッダから言語コードを取り出し、レスポンス文言を切り替えるため。

このプロジェクトは UI / メッセージともに日本語のみのため不要。

#### 5. テナント識別（マルチテナント）

**何のために**：サブドメインや Header から「どの組織のユーザーか」を判定するため。

このプロジェクトは単一テナント（個人ユーザー単位のみで、組織アカウント概念が無い）のため不要。

#### 6. WebSocket セッション管理

**何のために**：WebSocket 接続のライフサイクル管理（接続・切断・再接続のフック）。

このプロジェクトは「ジョブ進捗確認」を HTTP polling（フロントが定期的に GET）で実装する予定で、WebSocket を使わないため不要。

#### 7. API レイヤのレスポンスキャッシュ取得

**何のために**：リクエスト前段で Redis を覗き、ヒットしたら DB を叩かずに早返しするため。

このプロジェクトは「問題は毎回違う」「提出は冪等性なし」のように **同じリクエストが連続する場面が少ない**ため、API レイヤのキャッシュは投資対効果が低い。
LLM 出力キャッシュは Worker 側で別途実装するので、Backend の deps/ には置かない。

## deps/ の機能まとめ（カテゴリ別）

| カテゴリ                  | 何のために                                                          | 例                                                      |
| ------------------------- | ------------------------------------------------------------------- | ------------------------------------------------------- |
| **資源の貸し出し**        | 開閉のあるリソースを安全に使い回すため                              | DB セッション / Redis 接続 / 一時ファイルハンドル       |
| **入力の取り出し + 検証** | HTTP 入力（Cookie / Header / クエリ）を Python オブジェクトに整える | ログインユーザー / リクエスト ID / ページネーション引数 |
| **ガード（早期 reject）** | 条件を満たさないリクエストを 401 / 403 / 429 で弾く                 | 認可チェック / レート制限 / メンテナンス中フラグ        |
| **依存の組み立て**        | 他の dep を組み合わせて高機能な dep を作る                          | `get_current_user` の中で `get_async_session` を使う等  |
| **テスト時の差し替え口**  | フェイクに置換できる構造を作る                                      | テストで「ログイン中ユーザー」を固定値に差し替え        |

## 置くファイル（予定）

- `auth.py`：`get_current_user` / `require_admin`
- `db.py`：`get_async_session`
- `pagination.py`：`pagination_params`
- `observability.py`：`get_correlation_id`
- `rate_limit.py`：slowapi 設定（必要なら）
