# observability/

## observability/ とは何か

アプリの動きを **外から監視するための「計測装置のセットアップ」** を集めるフォルダ。
R5 / R6 で本格導入予定。

ここに置くのは「装置の組み立て・初期化処理」だけで、業務コードからは触らない。
`main.py` がアプリ起動時に 1 回だけ呼び、以降は裏で勝手に動く。
業務コード側はただ普通に `logger.info(...)` でログを書いたり、関数を呼んだりすれば、
ここでセットアップした仕組みが **自動で trace_id を付けたり Loki に流したり Sentry に送ったり**してくれる。

## なぜ必要か：observability の 3 つの観測軸（Three Pillars）

本番運用で「何が起きているか」を知るには、3 種類のデータを揃える必要がある。
このプロジェクトはこれを以下のスタックで実現する（[ADR 0041](../../../../docs/adr/0041-observability-stack-grafana-and-sentry.md)）：

| 観測軸 | 何を見るか | 採用プロダクト |
|---|---|---|
| **ログ（Logs）** | 「いつ・何が起きたか」のテキスト記録 | Grafana **Loki** |
| **メトリクス（Metrics）** | 「数値の時系列」（リクエスト数 / レイテンシ / エラー率 等） | **Prometheus** + Grafana |
| **トレース（Traces）** | 「1 リクエストが API → DB → Worker と渡る流れ」 | Grafana **Tempo** |
| **例外監視（Errors）**（補助） | 「例外発生時の詳細スタック + 通知」 | **Sentry** |

`observability/` フォルダは、これらに繋ぐ初期化コードを置く場所。

## observability/ で扱える機能パターン一覧

### 🟢 このプロジェクトで使うパターン

#### 1. 構造化ログ（JSON ログ）

**何のために**：ログを「あとから検索・集計できる形」で残すため。

ログをただの文字列で出すと、調査時には `grep` で検索するしかなく、
「特定ユーザーのエラーログだけ」「処理時間 1 秒超のリクエストだけ」のような
条件絞り込みが事実上不可能になる。ログをキー : 値ペアの JSON 1 行として出しておけば、
Loki 側で「`user_id = X` かつ `level = ERROR`」のような問い合わせが SQL 風にできる。

実態としては、ユーザーから「エラーが出ました」と問い合わせが来た時に
**該当ユーザーの該当時刻のログだけを 10 秒で出せる**ようにするのが目的。

#### 2. trace_id / correlation_id の自動付与

**何のために**：1 つのユーザー操作が複数サービスを横断する時、それを 1 本の糸として追跡するため。

このプロジェクトでは、ユーザーが「採点」ボタンを押すと **Frontend → API → DB（jobs テーブル）→ 採点 Worker → LLM 呼び出し → DB 書き戻し** と処理が連鎖する。
途中でエラーが起きた時、ログを別々に時刻で突き合わせて追うのは現実的でない。

リクエストの最初の地点で 1 つ ID を採番し、そのリクエストから派生する**全てのログ・トレース・例外通知**に
同じ ID を埋め込んでおけば、Loki / Tempo / Sentry のどこを開いても **ID 1 つで関連情報が全部出る**。
「ユーザーが 10:23 に押した採点ボタンの一連の挙動を全部見せて」を 1 検索で実現するのが目的。

#### 3. リクエスト / レスポンスのロギング

**何のために**：ユーザーから報告が来る前に、能動的に異常を見つけるため。

全 HTTP リクエストの「メソッド・パス・ステータスコード・処理時間」を自動記録しておくと、
「直近 1 時間で 5xx が増えていないか」「特定エンドポイントが急に遅くなっていないか」を
ダッシュボードで監視できる。

これが無いと「ユーザーが体験した不調」が**問い合わせが来るまで分からない受け身運用**になる。
記録があれば、デプロイ直後にエラー率が上がっていないかを能動的に確認できるし、
バグの影響範囲（何件のユーザーが踏んだか）を後から数えられる。

#### 4. Prometheus メトリクス出力

**何のために**：アプリ全体の健康状態を「数値の時系列」で把握し、閾値超えで自動アラートを出すため。

ログは「個別事象」、メトリクスは「全体傾向」を扱う担当。
「先週よりリクエスト数が 2 倍になった」「平均レイテンシが 200ms → 800ms に悪化した」のような
傾向は、ログを 1 件ずつ見ても気付けない。数値を時系列で蓄積して初めて見える。

このプロジェクトでは特に「LLM 呼び出しが詰まり始めていないか」「DB コネクションが枯渇していないか」のような
**じわじわ進行する障害**を早期検知する役割。閾値超えで Slack に通知も出せる。

#### 5. OpenTelemetry トレース（分散トレース）

**何のために**：「このリクエストは何処で時間を食っているか」を関数レベルで特定するため。

「採点 API が遅い」とログから分かっても、内訳が「DB クエリが 3 秒」「LLM 呼び出しが 8 秒」「JSON 変換が 0.1 秒」の
どれなのかは、コードを読んで自前で計測しないと分からない。
分散トレースは 1 リクエストの処理を**滝グラフ**（処理の入れ子構造）として記録しておくので、
本番のリクエスト 1 本を Grafana で開くだけで「どこに何 ms かかったか」が一目で見える。

このプロジェクトでは LLM 呼び出しがボトルネックになりやすいので、
「LLM のせいなのか / 自前コードのせいなのか」を**推測ではなく実測で切り分ける**のが目的。

#### 6. Sentry への例外送信

**何のために**：本番で起きた例外に即気付き、原因をその場で再現せずに特定するため。

ハンドルされていない例外（500 エラー）が起きた時、ログを掘り返してスタックトレースを探し、
リクエスト情報を別途突き合わせて、ローカルで再現実験して……の流れは時間がかかる。

Sentry は例外発生の**瞬間にメール / Slack 通知**を出し、スタックトレース・リクエスト情報・ユーザー ID 込みで
**1 画面に全て表示**する。類似エラーは自動でグルーピングされるので
「これ何件発生してる？影響ユーザーは？」も一覧で見える。
障害対応の「気付くまでの時間」と「原因特定までの時間」を縮めるのが目的。

#### 7. ジョブキュー深さのメトリクス

**何のために**：バックグラウンド処理の詰まりを早期検知し、ユーザーが「結果が返ってこない」と感じる前に手を打つため。

このプロジェクトの採点・問題生成は **LLM 呼び出しが外部 API 依存**で遅延が起きやすい。
未処理ジョブが溜まり続けると、ユーザー体験として「採点ボタン押したのに 5 分待っても結果が出ない」になる。

`jobs` テーブルの未処理件数を定期的に取得して Prometheus に流しておけば、
**「閾値を超えたら警告」「ある程度溜まったら Worker を自動増設」**のような対応が可能になる。
このプロジェクトで最も重要な健康指標の 1 つ。

#### 8. ヘルスチェック（`/healthz` / `/readyz`）

**何のために**：自動運用基盤に「このインスタンスは今リクエストを受けられるか」を機械可読な形で伝えるため。

Docker / Kubernetes / ロードバランサは、定期的にこのエンドポイントを叩いて返ってくる応答で
「このインスタンスにユーザーリクエストを流していいか」を判定している。

これが無いと、起動中・DB 接続失敗中のインスタンスにもユーザーリクエストが流れて
500 エラーを返してしまう。`/healthz`（プロセス生存）と `/readyz`（依存サービス接続 OK）を分けることで、
**準備ができるまで自動でトラフィックから除外**される。デプロイ時の無停止切替を可能にする土台。

> **配置の注意**：実装ファイルは [routers/probes.py](../routers/) + 依存サービスチェックを [deps/](../deps/) に置く。observability/ は「観測軸」の文脈でこの機能を扱うが、コード自体はここに置かない（実装は既に routers/probes.py に存在）。

### 🔴 このプロジェクトでは使わなさそうなパターン

参考までに「observability で書けるけどこのプロジェクトでは出番がない」パターンも挙げておく。

#### 1. Datadog / New Relic / Honeycomb 等の商用 APM

サブスクリプション課金の SaaS。
このプロジェクトは **Grafana スタック**（R0〜R5 は Grafana Cloud 無料枠、R6 以降は AWS 上で自前運用、[ADR 0041](../../../../docs/adr/0041-observability-stack-grafana-and-sentry.md)）でコストを抑える方針のため不採用。

#### 2. PagerDuty / Opsgenie 連携

夜間オンコール呼び出しサービス。
このプロジェクトは個人ポートフォリオで SLA も無いため、Sentry のメール / Slack 通知で十分。

#### 3. ユーザー行動分析（Mixpanel / Amplitude）

「どのボタンを何回押したか」を追跡してプロダクト改善に使う。
このプロジェクトはポートフォリオ用途で、行動分析よりもまず動くものを作る方が優先。
将来必要なら Frontend 側に別途追加する話で、Backend の observability/ には置かない。

#### 4. プロファイラ（Pyroscope / py-spy）

「どの関数で CPU 時間を食っているか」を継続的に観測する。
このプロジェクトの規模では Prometheus のレイテンシヒストグラムで十分で、関数単位のプロファイラまでは不要。

#### 5. eBPF ベースの可視化（Pixie / Cilium Hubble）

カーネルレベルでネットワーク・syscall を観測する仕組み。
このプロジェクトは単一 EC2 / コンテナ規模で、ネットワーク詳細は不要。

#### 6. セッションリプレイ（Sentry Replay / LogRocket）

ユーザーのブラウザ操作を録画して再生する。
このプロジェクトは Backend 主体で、UI バグ調査の優先度が低いため見送り。

#### 7. RUM（Real User Monitoring）の Backend 統合

ブラウザ側のパフォーマンス計測データを集約する。
このプロジェクトでは Frontend 側に Sentry Browser SDK を入れれば完結し、Backend の observability/ で扱う話ではない。

#### 8. CloudWatch / Stackdriver 等のクラウド標準ログ

AWS / GCP 標準のログ・メトリクス機構。
このプロジェクトは Grafana スタックに集約するため、CloudWatch には流さない（コスト・ベンダーロックイン回避）。

#### 9. 分散ロギング以外の ELK / Splunk

Elasticsearch + Logstash + Kibana の組み合わせ。
Loki と機能が被り、Loki の方が軽量で本プロジェクト規模に合うため不採用。

## observability/ の機能まとめ（カテゴリ別）

| カテゴリ | 中身 | 例 |
|---|---|---|
| **ログ整形** | テキストログを構造化（JSON 化）して、検索・集計可能にする | json formatter / trace_id 注入 |
| **メトリクス出力** | 数値の時系列を `/metrics` エンドポイント経由で Prometheus に渡す | リクエスト数 / レイテンシ / キュー深さ |
| **トレース送信** | 「1 リクエストが渡る流れ」を Tempo に送る | OpenTelemetry instrumentation（FastAPI / SQLAlchemy / httpx） |
| **例外監視** | ハンドルされていない例外を別系統に送って即気付ける | Sentry SDK の `init` + middleware |
| **計測の伝播** | 1 つの ID を「ログ・スパン・例外」全てに付けて横断追跡 | `contextvars` ベースの trace_id 注入 |
| **ヘルスチェック** | 「アプリが生きているか / DB に繋がるか」を返す | `/healthz` / `/readyz` |

## 置くファイル（予定）

- `logging.py`：構造化ログ（JSON）の formatter 設定 + trace_id 注入
- `tracing.py`：OpenTelemetry の初期化（FastAPI / SQLAlchemy / httpx の instrumentation）
- `metrics.py`：Prometheus エクスポータの初期化 + `/metrics` ルート登録
- `sentry.py`：Sentry SDK の初期化（DSN・サンプリングレート設定）
- `middleware.py`：リクエスト / レスポンスのロギングミドルウェア

採用スタック（Grafana / Loki / Tempo / Prometheus + Sentry）の選定理由は
[ADR 0041](../../../../docs/adr/0041-observability-stack-grafana-and-sentry.md) を参照。
